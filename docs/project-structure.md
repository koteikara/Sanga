# project-structure.md

## 目的

このドキュメントは、京都サンガF.C.日程表HTMLプロジェクトの現在のファイル構成、データ構成、確認体制、CI構成を整理し、次の実装フェーズを安全に選びやすくするためのメモです。

今回の整理は、次の作業へ進む前の現状把握を目的とします。

今後の開発予定、完了済み機能、次に取り組む候補の全体整理は `docs/roadmap.md` を参照してください。

- CSS整理
- JavaScript整理
- HTML内に残っている構造の整理
- 日程データ更新フローの整理
- Googleスプレッドシート連携
- 本番アップロード手順の整理
- 実ブラウザ確認の自動化検討

このドキュメント追加では、公開HTML、CSS、JavaScript、日程JSON、GitHub Actionsワークフローの実装修正は行いません。
確認できない仕様や、今後の判断が必要な事項は「要確認」として扱います。

## 現在の主要ファイル

| 種別 | ファイル | 現在の役割 |
| --- | --- | --- |
| 公開HTML | `public/sanga202627season.html` | 公開ページ本体。CSSとJavaScriptを外部ファイルとして読み込み、日程表UIの土台になる。 |
| CSS | `public/assets/style.css` | スマートフォン優先のカードUI、表示列、ダイアログ、状態表示などの見た目を担当する。 |
| JavaScript | `public/assets/app.js` | 日程JSON読み込み、カード表示、操作状態、表示列、フィルタ、表示モード、共有画像関連などの画面挙動を担当する。 |
| 日程JSON | `public/data/matches.json` | 当面の公開表示で使う日程データ。将来のスプレッドシート連携・生成フローの入力または比較対象になる。 |
| 検証ツール | `tools/validate-matches.js` | 現在の公開JSONとHTMLの整合性を確認する。 |
| 生成JSON検証 | `tools/validate-generated-matches.js` | CSVなどから生成したJSON、または公開JSONをスキーマ・件数・値の観点で検証する。 |
| CI | `.github/workflows/static-checks.yml` | Pull Requestおよび`main`へのpush時に、静的検証を自動実行する。 |
| ドキュメント | `docs/` | 設計、運用、データスキーマ、確認方針、AI作業ログを管理する。 |

## public配下

### public/sanga202627season.html

現在の公開ページ本体です。

主な役割は次の通りです。

- ページの基本HTML構造を持つ。
- `assets/style.css` を読み込む。
- `assets/app.js` を`type="module"`で読み込む。
- JavaScript無効時やデータ読み込み前後の表示土台を提供する。
- 免責事項、使い方、保存内容削除など、利用者に見える説明・操作の入口を持つ。

変更時に特に注意が必要な点は次の通りです。

- 既存のDOM構造、class名、id名はJavaScriptやCSSから参照されている可能性がある。
- LocalStorageに保存された既存状態との互換性に影響する可能性がある。
- 免責事項、使い方、LocalStorage説明文は、利用者向け説明として重要なため、安易に簡略化しない。
- 日程、節番号、対戦相手、会場、キックオフ時刻などの事実情報は、明示的な依頼なしに変更しない。

今回の作業では、このファイルは変更しません。

### public/assets/style.css

公開ページの見た目を担当するCSSです。

主な役割は次の通りです。

- スマートフォン優先のカード型日程表レイアウトを定義する。
- 試合カード、ホーム・アウェイ表示、日付、会場、注記などの見た目を定義する。
- 表示列切替、コンパクト表示、リスト表示などの表示モードに関係する見た目を定義する。
- 使い方ダイアログ、フッター操作、免責事項などのUI表示を定義する。
- 対戦相手ロゴや状態表示など、データ表示と結びついた装飾を含む。

変更時に特に注意が必要な点は次の通りです。

- 同じセレクタの上書きや表示モードごとの差分がある可能性があるため、並び替えや統合は表示崩れにつながる場合がある。
- スマートフォン幅での閲覧性を優先する。
- フォーカス表示を消さない。
- 色だけに依存した状態伝達を増やさない。
- 日程カードの高さ、縦帯、日付欄、ボタン操作領域に影響する変更は実ブラウザ確認が必要。

今回の作業では、このファイルは変更しません。

### public/assets/app.js

公開ページの挙動を担当するJavaScriptです。

主な役割は次の通りです。

- `public/data/matches.json` を読み込み、日程データを画面に反映する。
- 試合カードの個人状態を管理する。
- 表示列、フィルタ、表示モードなどの表示設定を管理する。
- 使い方ダイアログ、LocalStorage削除、共有画像保存などの操作を扱う。
- LocalStorageへ保存した設定を読み込み、画面へ復元する。

変更時に特に注意が必要な点は次の通りです。

- LocalStorageキー、保存形式、復元処理を変更すると、既存利用者の保存状態に影響する。
- DOMのclass名、id名、data属性に依存している可能性がある。
- JSONの項目名やデータ欠損時の扱いを変える場合は、検証ツールと表示崩れの両方を確認する必要がある。
- ダイアログ、ボタン、表示列変更、LocalStorage削除はキーボード操作できる状態を保つ。

今回の作業では、このファイルは変更しません。

### public/data/matches.json

当面の公開表示で使う日程JSONです。

主な役割は次の通りです。

- `meta` にシーズン、チーム、更新日時、出典を持つ。
- `matches` に試合ごとのID、シーズン、大会、節、開催日または候補日、ホーム/アウェイ、対戦相手、会場、状態、注記などを持つ。
- Googleスプレッドシート連携やCSV生成・再生成フローの基準データとして扱われる。
- 既存HTMLとの整合性確認の対象になる。

変更時に特に注意が必要な点は次の通りです。

- 日程、節番号、対戦相手、会場、キックオフ時刻は、明示的な依頼なしに変更しない。
- `id` は試合カード状態の保存と結びつく可能性があるため、安易に変更しない。
- `status`、`date_candidates`、`note` など未確定情報の表現は、表示と検証の両方に影響する。
- 公式情報などを元に更新する場合は、可能な限り出典URLと確認日を残す。

今回の作業では、このファイルは変更しません。

## tools配下

`tools/` には、日程データの生成、検証、確認用出力に関するNode.jsスクリプトがあります。
この節では、今回の対象であるvalidate系ツールを中心に整理します。

### validate-matches.js

`public/data/matches.json` と `public/sanga202627season.html` を対象にした、現在の公開データ向け検証ツールです。

主な役割は次の通りです。

- `public/data/matches.json` をJSONとして読み込めるか確認する。
- `matches` の件数やID、必須項目、日付形式、状態値などを確認する。
- 現在のHTMLと公開JSONの主要情報が大きくずれていないかを確認する。

要確認事項は次の通りです。

- 今後HTMLの構造整理やJSON表示への完全移行を進める場合、このツールがどこまでHTML比較を続けるべきかは要確認。
- カップ戦、ACL、延期、中止、試合終了などの運用が増えた場合、許容値の拡張が必要になる可能性がある。

### validate-generated-matches.js

生成済みJSONを検証するためのツールです。

主な役割は次の通りです。

- 引数で指定したJSONファイルを検証する。
- `--expected-count` により件数を確認する。
- `--strict` により、必須項目や空値の扱いをより厳しく確認する。
- CSVやGoogleスプレッドシート由来の生成JSONを公開前に確認する。

要確認事項は次の通りです。

- Googleスプレッドシート連携が進んだ後、どの列を必須にするかは運用方針に合わせて見直しが必要。
- キックオフ時刻、チケットURL、放送配信、結果など、現時点で空欄があり得る項目の厳格度は要確認。

## GitHub Actions

### .github/workflows/static-checks.yml

Pull Requestおよび`main`ブランチへのpush時に実行される静的検証ワークフローです。

現在の主な役割は次の通りです。

- リポジトリをチェックアウトする。
- Node.js 20を設定する。
- `git diff --check` で空白エラーを確認する。
- `node tools/validate-matches.js` で日程JSONとHTMLの基本検証を行う。
- `node tools/validate-generated-matches.js public/data/matches.json --expected-count 49 --strict` で公開JSONを厳格検証する。
- `node --check public/assets/app.js` でJavaScript構文を確認する。
- PythonインラインスクリプトでCSSの波括弧数を確認する。
- PythonインラインスクリプトでHTMLがCSS/JavaScriptを参照していることを確認する。

このワークフローは、外部依存の少ない基本的な静的検証を担います。
実ブラウザでの表示崩れ、操作確認、アクセシビリティ確認までは保証しません。

今回の作業では、このファイルは変更しません。

## LocalStorage関連

現在のJavaScriptでは、LocalStorageを使って閲覧者ごとの状態や表示設定を保存しています。
確認できた主なキーは次の通りです。

| キー | 主な用途 |
| --- | --- |
| `sanga-schedule-button-states-v1` | 試合カードごとの個人状態保存。 |
| `sanga-schedule-filter-settings-v1` | フィルタ設定保存。 |
| `sanga-schedule-display-mode-v1` | 表示モード保存。 |
| `sanga-schedule-layout-v1` | 表示列設定保存。 |

変更時に特に注意が必要な点は次の通りです。

- 既存キー名は、明示的な移行方針なしに変更しない。
- 保存値の形式を変える場合は、旧形式からの移行または後方互換を検討する。
- 試合IDを変更する場合、試合カードごとの保存状態が復元できなくなる可能性がある。
- LocalStorage削除ボタンの対象範囲を変える場合は、利用者向け説明文も確認する。

キーの詳細な保存形式、削除対象、復元順序は、今後JavaScript整理時に改めて要確認です。

## 現在の確認体制

### 自動確認

現在の自動確認は、主にGitHub ActionsのStatic Checksとローカル実行可能なNode.jsスクリプトで構成されています。

主な確認内容は次の通りです。

- Git差分の空白エラー確認。
- `public/data/matches.json` の検証。
- 生成JSON形式の厳格検証。
- `public/assets/app.js` の構文確認。
- `tools/validate-app-contract.js` によるLocalStorageキー、JSON読み込みパス、主要DOM/CSSフック、表示列・表示モード・フィルタ有効値の静的契約確認。
- `public/assets/style.css` の波括弧数確認。
- `public/sanga202627season.html` のCSS/JavaScript参照確認。

自動確認でカバーできていない主な範囲は次の通りです。

- 実ブラウザでの表示崩れ確認。
- スマートフォン幅、PC幅、中間幅での目視確認。
- 表示列変更、使い方ダイアログ、LocalStorage削除などの操作確認。
- キーボード操作、フォーカス表示、スクリーンリーダー向けの詳細確認。
- 本番サーバー反映後の公開URL確認。

### 人間による軽量確認

人間による軽量確認は、`docs/ai/BROWSER_CHECKLIST.md` の5分スモークチェックを基準にします。

主な確認項目は次の通りです。

- スマートフォン幅で主要表示が大きく崩れていないこと。
- PC幅で主要表示が大きく崩れていないこと。
- 高さが大きい試合カードを含む行で、背景・縦帯・日付欄が不自然に短く見えないこと。
- 表示列変更が動くこと。
- 使い方ダイアログが開閉できること。
- LocalStorage削除ボタンが動くこと。
- ブラウザコンソールに赤いJavaScriptエラーが出ていないこと。

UIやJavaScriptの実装修正を行った場合は、静的検証だけでなく、人間による実ブラウザ確認が必要です。

## 変更しやすい範囲

比較的変更しやすい範囲は次の通りです。

- 現状整理、運用整理、確認手順などのドキュメント追加。
- `docs/ai/WORKLOG.md` への作業記録追記。
- 既存仕様を変えない範囲での説明追加。
- 検証コマンドや確認手順の明文化。
- 実装に影響しない調査メモの追加。

実装ファイルに進む場合でも、比較的安全に始めやすい候補は次の通りです。

- CSSの小さな重複整理。ただし表示差分確認が必要。
- JavaScript内のコメント追加や責務整理のための小さな抽出。ただしLocalStorage互換性確認が必要。
- データ更新手順のドキュメント化。

## 変更に注意が必要な範囲

特に注意が必要な範囲は次の通りです。

- `public/sanga202627season.html` のDOM構造、class名、id名、文言。
- `public/assets/style.css` のレイアウト、表示モード、フォーカス表示、ダイアログ関連。
- `public/assets/app.js` のLocalStorageキー、保存形式、復元処理、DOM参照、JSON読み込み処理。
- `public/data/matches.json` の試合ID、日程、節番号、対戦相手、会場、キックオフ時刻、状態値。
- `.github/workflows/static-checks.yml` の必須チェックや実行条件。
- 本番アップロードに関わるワークフロー、Secrets、認証情報の扱い。
- 公式サイト等の公開情報を参照するデータ更新。

これらを変更する場合は、変更前に目的、影響範囲、確認方法、戻し方を整理する必要があります。

## 今後の改善候補

今後の改善候補は次の通りです。

- CSSの整理
  - 表示モード、カードUI、ダイアログ、レスポンシブ指定などのまとまりを把握する。
  - まずは削除や並び替えをせず、重複や依存関係の調査から始める。
- JavaScriptの整理
  - 日程データ読み込み、カード描画、LocalStorage、フィルタ、表示モード、共有画像関連の責務を整理する。
  - 既存キーと保存形式を維持しながら、小さな単位で関数整理する。
- HTML内に残っている構造の整理
  - JavaScript生成部分と静的説明部分の役割を整理する。
  - 免責事項、使い方、LocalStorage説明文は維持し、変更時は文言の目的を確認する。
- 日程データ更新フローの整理
  - `matches.json` を更新する手順、検証する手順、差分確認する手順を明文化する。
  - 出典URL、確認日、未確定情報の扱いを運用手順に落とし込む。
- Googleスプレッドシート連携
  - 現在のCSV生成・JSON生成ツールを前提に、手動CSV運用から始める。
  - Google Sheets APIや認証情報を使う自動連携は、認証情報管理方針を決めてから検討する。
- 本番アップロード手順の整理
  - スターレンタルサーバーへの手動または手動実行ワークフローによる反映手順を確認する。
  - 本番反映前後の確認項目、ロールバック、キャッシュ更新方針を整理する。
- 実ブラウザ確認の自動化検討
  - Playwright等による表示・操作スモークチェックを検討する。
  - ただし外部依存や実行環境の制約があるため、まずは人間の5分確認を安定運用する。

## 次に進むおすすめ順

次に進むおすすめ順は次の通りです。

1. 本体を壊さないドキュメント整理
   - 現状構成、更新手順、確認手順、運用判断を整理する。
   - 公開HTML/CSS/JavaScript/JSONを変更しないため、リスクが低い。
2. 小さなCSS整理
   - 表示崩れリスクを抑えながら、重複や責務の見える化から始める。
   - 変更後はスマートフォン幅とPC幅で軽量確認する。
3. 小さなJavaScript整理
   - LocalStorageキーと保存形式を維持したまま、責務単位で小さく整理する。
   - 変更後は表示列、フィルタ、ダイアログ、削除ボタンの操作確認を行う。
4. データ更新フロー整理
   - CSV、JSON生成、検証、差分確認、本番反映前確認の流れを整える。
   - 未確定情報、出典、確認日の扱いを明確にする。
5. 自動更新・デプロイ検討
   - Googleスプレッドシート連携、実ブラウザ確認自動化、本番アップロード自動化を段階的に検討する。
   - 認証情報はリポジトリに保存せず、GitHub Secrets等で管理する方針を維持する。
