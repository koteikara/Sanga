# 表示モード切替の設計

## この設計の目的

年間スケジュールページで、利用場面に応じた見やすい表示を選べるようにするため、表示モード切替の考え方を整理します。

表示モード切替は、スマートフォンでの日程確認、フィルタ結果の確認、フルページスクショ用表示、SNS投稿補助、パーソナライズ状態の見せ方、将来的な印刷・PDF向け表示、チケット・観戦準備メモとの連携に向けた土台として扱います。

今回の文書は設計方針の整理のみを目的とし、HTML / CSS / JavaScript の実装は行いません。

## 現在の表示仕様

現在の年間スケジュールページは、JSON由来の日程カードを表示しています。ページ内には年見出し `2026 / 2027` があり、日程カードを年単位で確認できます。

表示列は `2列 / 3列 / 4列` で切り替えられます。表示列設定はLocalStorageに保存され、再読み込み後も利用者の選択が復元されます。

表示列の変更は設定パネル内で行います。フィルタリングも設定パネル内で行い、日程表示欄の上部には操作UIを増やさない方針です。

日程カードはタップすることで、赤色枠・水色枠の状態を切り替えられます。赤色枠・水色枠の意味はページ側で固定せず、利用者が自由に決める方針です。

## 表示モード切替の基本方針

日程表示欄は、できるだけシンプルに保ちます。表示モードの操作は、日程表示欄の上部へ常時追加するのではなく、設定パネルに集約します。

初期表示は現在のカード表示を維持します。既存の `2列 / 3列 / 4列` 切替は壊さず、表示列切替と表示モード切替は別概念として整理します。

表示モードは、フィルタリング結果にも適用できるようにします。表示モードを切り替えても、カード状態やフィルタ条件は消さない方針です。

スマートフォンでの見やすさを最優先にします。あわせて、SNS投稿やスクショ用表示への展開を見据えます。

## 初期実装候補

### 通常カード表示

通常カード表示は、現在のカード表示です。試合情報を視覚的に見やすく表示し、`2列 / 3列 / 4列` 切替と組み合わせます。

初期状態として維持し、既存利用者が違和感なく使い続けられる表示モードにします。

### コンパクト表示

コンパクト表示は、カードの縦幅や装飾を抑え、多くの試合を一度に見たい場合に使う表示です。

スマートフォンでスクロール量を減らすことを目的にします。ただし、日付、対戦相手、会場、状態などの重要情報を削りすぎないようにします。

### リスト表示

リスト表示は、1試合1行またはそれに近い形で表示するモードです。日付、H/A、対戦相手、会場、状態を一覧しやすくします。

フィルタ結果の確認に向いています。一方で、視覚的なカード感は弱くなるため、通常カード表示とは用途を分けます。

### スクショ用表示

スクショ用表示は、SNS投稿や共有画像向けに見やすく整える表示です。選択中フィルタや色枠状態を反映できるようにします。

余計な操作ボタンや説明文は隠す想定です。作成者表記や免責事項をどう扱うかは、実装前に検討します。

アクセシビリティ上、画像投稿だけに依存しない注意も必要です。投稿本文や代替テキストと組み合わせて使う前提にします。

## 将来拡張候補

### 月別表示

月ごとに試合を区切り、短い期間の予定を確認しやすくする表示です。遠い月の試合まで一度に表示しすぎないため、スマートフォンでの見通し改善にもつながります。

### ホームのみ強調表示

HOME試合を視覚的に強調し、ホーム観戦中心の利用者が予定を把握しやすくする表示です。AWAYを非表示にするのではなく、強調度の差として扱う案も検討します。

### 遠征向け表示

AWAY試合、会場、キックオフ時刻、移動や宿泊の検討に関係する情報を見やすくする表示です。将来的には観戦準備メモとの連携も考えられます。

### チケット確認表示

チケットURL、販売状況、購入済み状態などを確認しやすくする表示です。個人の購入状況を扱う場合は、LocalStorageなど端末内保存を基本にします。

### 観戦準備メモ表示

持ち物、移動、集合時間、観戦メモなどを試合ごとに確認しやすくする表示です。公開JSONではなく、利用者ごとのローカルな情報として扱うことを前提にします。

### 結果・戦績表示

試合後の得点、勝敗、戦績を確認しやすくする表示です。年間の進行状況を振り返る用途に向いています。

### 印刷・PDF向け表示

紙やPDFで確認しやすいように、余白、文字サイズ、不要な操作UIの非表示を調整する表示です。スクショ用表示とは別に、印刷媒体向けの可読性を重視します。

## 設定パネルとの関係

表示モード切替は設定パネル内に置きます。日程表示欄の上部にモード切替ボタンを常時並べない方針です。

設定パネル内では、`表示` 区分に追加するのが自然です。既存の `2列 / 3列 / 4列` は表示列設定として維持し、表示モードは表示列設定とは別項目として扱います。

構成例は次のとおりです。

```text
設定
├─ 表示
│  ├─ 表示列
│  │  ├─ 2列
│  │  ├─ 3列
│  │  └─ 4列
│  └─ 表示モード
│     ├─ 通常カード
│     ├─ コンパクト
│     ├─ リスト
│     └─ スクショ用
├─ 絞り込み
└─ 保存状態
```

## フィルタリングとの関係

表示モードは、現在のフィルタ結果に対して適用します。フィルタ条件を変えても表示モードは維持し、表示モードを変えてもフィルタ条件は維持します。

フィルタ結果が0件のときのメッセージは、どの表示モードでも表示します。年見出しの表示・非表示制御も維持します。

スクショ用表示では、フィルタ結果だけを対象にするか、赤色枠・水色枠などの選択状態だけを対象にするかを検討します。

## パーソナライズ状態との関係

赤色枠・水色枠の意味は固定しません。利用者が自由に意味を決める方針を維持します。

表示モードによって、色枠状態の見せ方を変えられます。リスト表示では、色枠を枠線だけでなくテキストやアイコンでも示す必要があります。

スクショ用表示では、色だけでなくラベルや凡例も併用します。将来、利用者が色枠ラベルを設定できるようになった場合は、そのラベルを表示モードに反映します。

## スクショ用表示との関係

スクショ用表示は、SNSで視覚的に共有しやすくするための表示です。フルページスクショ用表示はSNS投稿機能の重要な土台として扱います。

操作ボタン、設定パネル、使い方パネルなどは非表示にする想定です。フィルタ結果や色枠状態は、スクショ用表示にも反映します。

見出し、対象期間、表示件数、作成者表記をどう扱うかは検討が必要です。画像だけで情報提供が完結しないよう、投稿文やテキスト補助もあわせて検討します。

アクセシビリティ面では、画像投稿時に代替テキストや本文併記を推奨する方針にします。

## SNS投稿補助との関係

スクショ用表示とSNS投稿補助は密接に関係します。投稿文生成では、フィルタ結果や選択状態を使えるようにする可能性があります。

投稿文には、試合数、対象年、HOME/AWAY、対戦相手などを含められます。画像投稿だけでなく、テキスト投稿も選択肢として残します。

X投稿向けに短い文面を生成する可能性があります。ただし、今回のPRでは投稿機能は実装しません。

## LocalStorage保存方針

既存のLocalStorageキーは変更しません。

```text
sanga-schedule-button-states-v1
sanga-schedule-layout-v1
sanga-schedule-filter-settings-v1
```

表示モード用の将来キー候補は次のとおりです。

```text
sanga-schedule-display-mode-v1
```

保存イメージは次のとおりです。

```json
{
  "mode": "card"
}
```

表示モード候補の値例は次のとおりです。

```text
card
compact
list
screenshot
```

表示モード設定は、表示列設定とは別キーにします。フィルタ条件とも別キーにします。

保存内容削除ボタンの対象に含めるかは、実装時に検討します。不正な値が保存されていた場合は、通常カード表示に戻します。既存のカード状態やフィルタ条件を壊さないことを優先します。

## アクセシビリティ上の注意

表示モード切替ボタンには `aria-pressed` を使います。設定パネル開閉の既存アクセシビリティを壊さないようにします。

色だけで状態を伝えません。リスト表示やコンパクト表示でも、情報の欠落に注意します。スクショ用表示でもテキスト情報を残します。

画像投稿時は、代替テキストや投稿本文で補う方針を示します。表示モード変更時に必要であれば、`aria-live` で状態変化を伝えます。

キーボード操作で表示モードを切り替えられるようにします。アニメーションや表示切替効果を入れる場合は、`prefers-reduced-motion` に配慮します。

## 実装時の注意

今回は設計のみで実装しません。実装時は、既存カード生成処理を大きく壊さないようにします。

表示モード切替は、まずCSSクラス切替で実現できるか検討します。リスト表示など、DOM構造を変える必要がある場合は慎重に行います。

既存のフィルタ処理、年見出し制御、表示件数表示を壊さないことを優先します。View Transitions APIやマイクロインタラクションを使う場合は、未対応環境フォールバックと `prefers-reduced-motion` を守ります。

HTML / CSS / JavaScript を変更する実装時は、キャッシュクエリとページ下部バージョン更新を忘れないようにします。`public/data/matches.json` は、表示モード実装では原則変更しません。

## やってはいけないこと

今回のPRでは、表示モード切替を実装しません。本番ページのHTML / CSS / JavaScript、公開JSON、GitHub Actions、依存関係ファイルは変更しません。

既存のLocalStorageキーは変更しません。赤色枠・水色枠の意味を固定しません。認証情報、APIキー、FTP情報、サービスアカウントJSON、Secretsの実値は作成・記載しません。

日程表示欄の上部に操作UIを増やしすぎないようにします。スクショ用表示を作る場合も、画像だけで情報提供が完結する設計にはしません。
