# JS_CHANGE_CHECKLIST.md

## 目的

`public/assets/app.js` を変更するPRで、既存の表示・保存・操作仕様を壊さないための軽量チェックリストです。

このチェックリストは、`docs/js-inventory.md` の棚卸し内容をもとに、JavaScript整理、関数分割、LocalStorage整理、JSON描画改善へ進む前後で確認すべき観点をまとめます。

毎回すべてを目視で細かく確認する前提にはせず、次の3種類に分けて使います。

- 自動確認で毎回見ること
- 人間が短時間で軽く確認すること
- 変更内容に関係する場合だけ追加で確認すること

## 使うタイミング

- `public/assets/app.js` を変更する前に、変更範囲と影響しそうな機能を確認する。
- `public/assets/app.js` を変更した後に、自動確認と軽量確認の結果をPRまたはWORKLOGへ記録する。
- LocalStorage、DOM参照、CSSクラス連動、表示列、表示モード、フィルタ、ダイアログ、JSON描画、共有画像関連に触る場合は、該当する追加確認も行う。

## 毎回必ず確認すること

- [ ] 今回の変更が `public/assets/app.js` に必要な変更かを説明できる。
- [ ] `node --check public/assets/app.js` が通る。
- [ ] Static Checks が通る、またはローカルで同等の静的確認を実行している。
- [ ] LocalStorageキー名を変更していない。
- [ ] HTMLのid/class/data属性参照を変更していない、または変更理由を記録している。
- [ ] CSSクラスの付与・削除を変更していない、または変更理由を記録している。
- [ ] `matches.json` の読み込みパスを変更していない、または変更理由を記録している。
- [ ] 変更対象外の共有画像関連に触れていない。

## 自動確認で見ること

- [ ] `git diff --check` で空白エラーがない。
- [ ] `node --check public/assets/app.js` でJavaScript構文エラーがない。
- [ ] `node tools/validate-matches.js` で公開JSONとHTMLの基本検証が通る。
- [ ] `node tools/validate-generated-matches.js public/data/matches.json --expected-count 38 --strict` で公開JSONの件数と形式が通る。
- [ ] Static Checks が実行できる場合は、GitHub Actionsまたはローカル同等確認の結果を記録する。

## JavaScript契約チェック

- [ ] `node tools/validate-app-contract.js` で、LocalStorageキー、JSON読み込みパス、主要DOM/CSSフックが維持されていることを確認する。
- [ ] この契約チェックは Static Checks でも自動実行される。
- [ ] この契約チェックは、LocalStorageキー、JSON読み込みパス、主要DOM/CSSフックの破壊を検知する補助であり、実ブラウザでの表示・操作確認の代替ではない。

## 人間が軽く確認すること

実ブラウザ確認は、毎回重くしすぎず `docs/ai/BROWSER_CHECKLIST.md` の「通常確認: 7項目」を参照します。

- [ ] スマホ幅とPC幅で主要表示が大きく崩れていない。
- [ ] 表示列変更が動く。
- [ ] 使い方ダイアログが開閉できる。
- [ ] LocalStorage削除ボタンが動く。
- [ ] ブラウザコンソールに赤いJavaScriptエラーが出ていない。

問題がなければ「7項目スモークチェック問題なし」のように短く記録し、問題が出た場合だけ再現手順や画面幅を残します。

## 変更内容別の追加確認

### LocalStorageに関係する変更

- [ ] 4つの既存キーを変更していない。
  - `sanga-schedule-button-states-v1`
  - `sanga-schedule-filter-settings-v1`
  - `sanga-schedule-display-mode-v1`
  - `sanga-schedule-layout-v1`
- [ ] 保存形式を変えた場合、旧形式の読み取り互換または移行処理がある。
- [ ] `match.id` と試合カードごとの保存状態の紐づきが維持されている。
- [ ] LocalStorage削除ボタンの削除対象と、画面上の説明文が一致している。
- [ ] LocalStorageを使えない環境でも、画面が大きく破綻しない。

- [ ] LocalStorage利用可否フラグ名を変更しても、キー文字列・保存形式・削除対象を変更していない。
- [ ] `node tools/validate-app-contract.js` で4つのLocalStorageキー文字列が残っていることを確認した。

### 表示列・表示モードに関係する変更

- [ ] 1列、2列、3列、4列の切替が動く。
- [ ] 再読み込み後に表示列設定が復元される。
- [ ] カード表示とコンパクト表示の切替が動く。
- [ ] 再読み込み後に表示モード設定が復元される。
- [ ] `aria-pressed` が選択状態と一致している。
- [ ] `data-layout`、`data-display-mode`、表示モード関連クラスの更新がCSSとずれていない。

### フィルタに関係する変更

- [ ] `all`、`home`、`away`、`year-2026`、`year-2027`、`tentative`、`marked`、`state-1`、`state-2` の扱いを壊していない。
- [ ] フィルタボタンの選択状態と `aria-pressed` が一致している。
- [ ] 件数表示が現在の絞り込み結果と一致している。
- [ ] 該当なしの場合の空メッセージが崩れていない。
- [ ] 年見出しの表示・非表示が絞り込み結果と大きくずれていない。

### 使い方ダイアログ・設定パネルに関係する変更

- [ ] 閉じる待機時間を整理する場合は、`PANEL_CLOSE_DELAY_MS` が従来どおり `240ms` を表している。
- [ ] 使い方ダイアログを開閉できる。
- [ ] 設定パネルを開閉できる。
- [ ] Escapeで閉じる。
- [ ] 背景オーバーレイ操作による開閉が意図どおりである。
- [ ] `aria-expanded` が開閉状態に合わせて更新される。
- [ ] `aria-hidden` や `hidden` の更新が表示状態とずれていない。
- [ ] フォーカス移動が大きく破綻していない。

### JSON読み込み・カード描画に関係する変更

- [ ] `matches.json` の読み込みパスとキャッシュ更新方針を不用意に変えていない。
- [ ] `matches.json` の件数38件が維持される。
- [ ] `match.id` とカード状態保存が連動する。
- [ ] 候補日、未定、注記、HOME/AWAY、会場が表示される。
- [ ] 対戦相手名、節、日付、曜日、年見出しが大きく崩れていない。
- [ ] データ欠損時にも、画面全体が破綻せず確認しやすい状態になる。

### 共有画像関連に関係する変更

- [ ] 今回触る必要があるかを、変更前にPRまたはWORKLOGへ記録している。
- [ ] CDN依存があるため、静的検証だけで完了扱いにしていない。
- [ ] 共有画像生成ボタンが動く。
- [ ] 保存リンクが表示され、保存導線が分かる。
- [ ] 終了ボタンで共有画像モードを終了できる。
- [ ] 生成中、成功、失敗の状態表示が分かる。
- [ ] フィルタ中や表示モード変更後でも、生成対象が意図どおりである。

## 変更してはいけないもの

明示的な依頼や移行方針がない限り、次のものは変更しません。

- LocalStorageの既存キー名。
- 保存済み表示設定、表示列、表示モード、フィルタ、カード状態の互換性。
- `matches.json` の読み込みパス。
- `match.id` とカード状態保存の紐づき。
- HTML側の既存id/class/data属性との連動。
- CSS側の表示列、表示モード、カード状態、共有画像関連クラスとの連動。
- 使い方ダイアログ、設定パネル、LocalStorage削除ボタンのキーボード操作性。
- 共有画像生成まわりの処理。ただし明示的に対象になっている場合を除く。

## JavaScript契約チェックの有効値確認

- [ ] `tools/validate-app-contract.js` は、表示列、表示モード、フィルタの有効値が `public/assets/app.js` に残っていることも静的に確認する。
- [ ] このチェックは静的な契約確認であり、表示列切替、表示モード切替、フィルタ操作、カード状態切替の実ブラウザ確認の代替ではない。

## LocalStorageキー定数名を変更する場合

- [ ] JavaScript内部の定数名を変更しても、LocalStorageキー文字列自体は変更しない。
- [ ] `node tools/validate-app-contract.js` で4つのLocalStorageキー文字列が残っていることを確認する。
