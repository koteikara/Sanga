# スプレッドシート更新から本番反映までの運用手順

## この手順の目的

この手順は、京都サンガF.C.年間スケジュールページの日程更新、キックオフ時刻追加、会場確定、チケットURL追加、放送配信情報追加、戦績更新などを、安全に本番反映するための標準手順です。

Googleスプレッドシートの `試合` シートを更新したあと、CSV出力、JSON生成、検証、PR作成、GitHub Pages確認、スターレンタルサーバーへの本番デプロイ、本番URL確認までを毎回同じ流れで実施します。

このPRでは本番デプロイは行いません。実際の本番反映は、GitHub Pagesで確認し、PRをマージしたあとに手動デプロイワークフローで行います。

## 全体の流れ

```text
Googleスプレッドシート
→ CSVダウンロード
→ docs/sheets/schedule.initial.csv へ反映
→ generate-matches-from-csv.js
→ public/data/matches.json
→ validate-generated-matches.js
→ validate-matches.js
→ PR作成
→ GitHub Pages確認
→ スターレンタルサーバーへ手動デプロイ
→ 本番URL確認
```

標準的な作業順は次のとおりです。

1. Googleスプレッドシートの `試合` シートを更新します。
2. `試合` シートだけをCSVでダウンロードします。
3. CSV内容を `docs/sheets/schedule.initial.csv` に反映します。
4. CSVから `public/data/matches.json` を生成します。
5. 生成JSONと公開用JSONを検証します。
6. 差分を確認してPRを作成します。
7. PRマージ後、GitHub Pagesで表示を確認します。
8. GitHub Actionsからスターレンタルサーバーへ手動デプロイします。
9. 本番URLで表示とキャッシュ状態を確認します。

## 1. Googleスプレッドシートを更新する

Googleスプレッドシートでは、年間スケジュールページ用の `試合` シートを更新します。列定義と入力ルールの詳細は `docs/sheets/schedule-columns.md` と `docs/sheets/schedule-template.md` を参照してください。

更新時は、次のルールを守ります。

* IDは `sec01`〜`sec38` を維持します。
* 表示順は1〜38を維持します。
* 開催日と候補日は同時入力しません。
* 開催日は `YYYY-MM-DD` 形式にします。
* 候補日は複数ある場合 `|` 区切りにします。
* 状態は `確定` / `未確定` のどちらかにします。
* 試合状態は `試合前` / `試合中` / `試合終了` / `延期` / `中止` / `未定` のいずれかにします。
* 表示フラグは `表示` / `非表示` のどちらかにします。
* `メモ` は運用者向け非公開メモであり、公開JSONには出しません。
* 個人メモ、参戦予定、視聴予定、気になる試合などはLocalStorage側で扱うため、スプレッドシートには入れません。
* チケット販売開始日時やイベント申込締切などは、年間スケジュールページの試合シートには詰め込まず、将来のニュース連動カレンダー側または別データで扱います。

## 2. 試合シートをCSVでダウンロードする

Googleスプレッドシートで `試合` シートを開いた状態で、次の操作を行います。

```text
ファイル
→ ダウンロード
→ カンマ区切り形式（.csv）
```

CSV出力時の注意点は次のとおりです。

* **現在開いているシートだけがCSV出力されます。** 必ず `試合` シートを開いた状態でダウンロードします。
* フィルタで一部行だけが表示されていないか確認します。
* ヘッダー行と38件の試合行が含まれているか確認します。
* `注記`、`設定`、`確認` シートはCSV出力対象ではありません。

## 3. リポジトリへCSV内容を反映する

ダウンロードしたCSVの内容を、リポジトリ側の `docs/sheets/schedule.initial.csv` に反映します。

反映時の注意点は次のとおりです。

* ヘッダー行は31列のまま維持します。
* 文字コードはUTF-8にします。
* `tmp/` 配下は一時ファイル置き場であり、コミットしません。
* スプレッドシートからダウンロードした元CSVをそのままコミットするのではなく、必要に応じて `docs/sheets/schedule.initial.csv` として管理します。
* 反映後は、意図しない列削除、列順変更、文字化け、改行崩れがないか確認します。

## 4. CSVからmatches.jsonを生成する

`docs/sheets/schedule.initial.csv` から公開用JSONを生成します。

```bash
node tools/generate-matches-from-csv.js docs/sheets/schedule.initial.csv public/data/matches.json
```

生成時の注意点は次のとおりです。

* `public/data/matches.json` を更新する前に、差分が想定内か確認します。
* `id` はLocalStorageの保存状態と紐づくため変更しません。
* `sec01`〜`sec38` を維持します。
* `メモ` 列を公開JSONへ出しません。
* `match_status`、`is_visible` などの拡張項目は追加されていてよいです。
* 画面側が参照する主要項目を欠落させません。
  * `id`
  * `round`
  * `home_away`
  * `home_away_label`
  * `opponent`
  * `opponent_code`
  * `venue`
  * `match_date`
  * `date_candidates`
  * `status`
  * `note`

## 5. 生成JSONを検証する

生成した `public/data/matches.json` を検証します。

```bash
node tools/validate-generated-matches.js public/data/matches.json --expected-count 49
node tools/validate-matches.js
```

スクリプトの構文確認も行います。

```bash
node --check tools/generate-matches-from-csv.js
node --check tools/validate-generated-matches.js
node --check tools/export-matches-to-sheet-csv.js
```

検証時は、次の観点も確認します。

* 38件あること。
* ID重複がないこと。
* `sec01`〜`sec38` が維持されていること。
* 日付形式が `YYYY-MM-DD` であること。
* 開催日と候補日が同時入力されていないこと。
* 候補日が複数ある場合、`date_candidates` が配列として出力されていること。
* `メモ` 列の内容が公開JSONに出ていないこと。
* `sec16` など未定扱いの試合が意図どおり表示できるデータになっていること。

## 6. GitHubへPRを作成する

検証に成功したら、差分を確認してGitHubへPRを作成します。

PR作成前に確認することは次のとおりです。

* `docs/sheets/schedule.initial.csv` の差分がスプレッドシート更新内容と一致していること。
* `public/data/matches.json` の差分がCSV生成結果として妥当であること。
* HTML/CSS/JavaScriptを変更していない場合は、それらのファイルに差分がないこと。
* GitHub Actionsワークフローや認証情報関連ファイルに不要な差分がないこと。
* `.env`、秘密鍵、サービスアカウントJSON、APIキー、Secretsの実値が含まれていないこと。
* `tmp/` 配下の生成物をコミットしていないこと。

PR本文には、更新内容、検証コマンド、GitHub Pagesで確認する予定の項目、本番デプロイは別途手動で行うことを記載します。

## 7. GitHub Pagesで確認する

PRをマージして `main` ブランチへ反映すると、GitHub Pages確認環境のワークフローが動きます。Actionsの `GitHub Pages確認環境` が成功したあと、GitHub Pages上の公開ページを確認します。

確認項目は次のとおりです。

```text
□ ページが正常に読み込まれる
□ 38件表示される
□ 2026 / 2027 の見出しが出る
□ sec16 が未定として表示される
□ ※1〜※7 の注記表示が崩れていない
□ カードをタップして赤色枠・水色枠が切り替わる
□ リロード後も枠線状態が残る
□ 保存内容削除ボタンでLocalStorage状態を削除できる
□ 2列 / 3列 / 4列切替が動く
□ スマホ表示で崩れない
□ ページ下部のバージョン情報が更新されている
```

## 8. スターレンタルサーバーへ本番デプロイする

GitHub Pagesで問題がないことを確認したあと、GitHub Actionsからスターレンタルサーバーへ手動デプロイします。

```text
GitHub
→ Sanga リポジトリ
→ Actions
→ スターレンタルサーバーへ手動デプロイ
→ Run workflow
→ branch: main
```

手動デプロイ時の注意点は次のとおりです。

* 本番デプロイは明示的に実施する場合のみ行います。
* ワークフローの `confirm` には、ワークフローで求められている確認文字列を入力します。
* デプロイ対象は `public/` 配下のみです。
* Secretsの実値はリポジトリやPR本文に書きません。
* デプロイ前に `main` ブランチへ反映済みであることを確認します。

## 9. 本番URLで確認する

本番URLで、GitHub Pages確認と同じ観点を再確認します。

```text
□ ページが正常に読み込まれる
□ 38件表示される
□ 2026 / 2027 の見出しが出る
□ sec16 が未定として表示される
□ ※1〜※7 の注記表示が崩れていない
□ カードをタップして赤色枠・水色枠が切り替わる
□ リロード後も枠線状態が残る
□ 保存内容削除ボタンでLocalStorage状態を削除できる
□ 2列 / 3列 / 4列切替が動く
□ スマホ表示で崩れない
□ ページ下部のバージョン情報が更新されている
```

追加で確認することは次のとおりです。

* スマホ実機で確認します。
* 古いCSSやJSONが残っている場合は、再読み込みまたはキャッシュ削除を試します。
* 一時確認としてURL末尾に `?s=check` のようなクエリを付けて確認してもよいです。

## 10. トラブル時の確認ポイント

### FTPの `server-dir should be a folder`

* `STAR_SERVER_REMOTE_DIR` はフォルダとして扱われるため、必要に応じて末尾 `/` を付けます。
* アカウントでログインしたディレクトリ直下に置く場合は、空欄またはワークフロー仕様に沿った値を使います。
* 既存ドキュメントに正しい設定があれば、それに合わせます。

### FTPの `530 Login incorrect`

次のSecretsまたは設定値を確認します。実値はリポジトリ、PR、Issue、ドキュメントに書きません。

* `STAR_SERVER_HOST`
* `STAR_SERVER_USER`
* `STAR_SERVER_PASSWORD`
* `STAR_SERVER_REMOTE_DIR`

### FTPの `Timeout (control socket)`

* 一時的な通信問題の可能性があります。
* 何も変更していない場合は再実行で成功することがあります。
* 連続して失敗する場合はFTP情報、サーバー状態、接続制限を確認します。

### アップロード成功なのに表示が変わらない

* `STAR_SERVER_REMOTE_DIR` が正しいか確認します。
* 意図しないディレクトリへアップロードしていないか確認します。
* 本番URLの設置場所とアップロード先が一致しているか確認します。
* ブラウザキャッシュを確認します。

### スマホだけ古い表示になる

* キャッシュの可能性が高いです。
* JSONクエリ、app.jsクエリ、ページ下部バージョンを確認します。
* ブラウザキャッシュ削除、再読み込み、クエリ付きURL確認を試します。

## 11. 更新種別ごとの注意点

### 日程更新

* 開催日と候補日を同時入力しません。
* 状態を `確定` / `未確定` で更新します。
* 根拠URLと確認日を更新します。

### キックオフ時刻追加

* `キックオフ時刻` は `HH:MM` 形式で入力します。
* 未定の場合は空欄にします。

### 会場確定

* `会場` を更新します。
* 必要に応じて `公開メモ` を更新します。

### チケットURL追加

* `チケットURL` に公式URLを入れます。
* チケット販売開始日時は試合シートには入れません。

### 放送配信追加

次の列を更新します。

* `放送配信`
* `放送配信URL`

### 戦績更新

次の列を更新します。

* `試合状態`
* `京都得点`
* `相手得点`
* `結果`

得点と結果が矛盾しないように確認します。試合終了後は、必要に応じて根拠URLと確認日も更新します。

## 12. やってはいけないこと

* 本番サーバーを直接編集しません。
* 本番デプロイをPR内や確認前に行いません。
* 認証情報、サービスアカウントJSON、APIキー、`.env`、Secretsの実値を作成・保存・記載しません。
* `tmp/` 配下の一時生成物をコミットしません。
* `id` を安易に変更しません。`id` はLocalStorageの保存状態と紐づきます。
* `メモ` 列を公開JSONへ出しません。
* 個人メモ、参戦予定、視聴予定、気になる試合をスプレッドシートに入れません。
* チケット販売開始日時やイベント申込締切を試合シートへ詰め込みません。
* CSSやJS自体を変更していない場合は、不要にCSS/JSのバージョンを変えません。
* HTML/CSS/JavaScriptの表示変更、Google Sheets API連携、ニュース抽出、カレンダー連携、SNS投稿機能、フィルタリング機能をこの手順更新と混ぜません。

### キャッシュ対策

JSON更新時は、必要に応じて次を確認・更新します。

* `public/assets/app.js` 内の `data/matches.json?v=...`
* `public/sanga202627season.html` の `assets/app.js?v=...`
* ページ下部のバージョン表記

現在の例は次のとおりです。

```text
Version 2026.06.19-1 / Data updated: 2026-06-19
```

CSSやJS自体を変更していない場合は、不要にCSS/JSのバージョンを変えない方針です。ただし、JSON読み込みキャッシュを更新するために `app.js` を変更した場合は、HTML側の `app.js` クエリも合わせて更新します。
