# スクショ用表示 / SNS投稿補助の設計

## この設計の目的

スクショ用表示とSNS投稿補助は、閲覧者が自分の参戦予定、気になる試合、絞り込み結果などをスマートフォンで見やすい形に整え、XなどのSNSへ共有しやすくするための将来機能です。

主な目的は次のとおりです。

* スマートフォンの縦長スクショで見やすい共有用表示を作る。
* フィルタ結果やカード状態をSNSで共有しやすくする。
* 「HOMEだけ」「AWAYだけ」「2026年分だけ」「未確定試合だけ」など、利用者の確認条件を投稿しやすくする。
* 赤色枠・水色枠を使った個人の整理状態を、視覚的に分かりやすく共有できるようにする。
* 画像として見やすく整えつつ、画像だけに依存せず、投稿文の生成やコピーも補助する。
* 非公式ページであること、正確な情報は公式情報を確認すべきことを維持する。

この文書では実装は行わず、将来実装する場合の方針、画面構成、保存方針、注意点を整理します。

## 現在の関連機能

現在の年間スケジュールページには、スクショ用表示とSNS投稿補助の土台になる機能がすでにあります。

* `public/data/matches.json` 由来の日程カード表示。
* 年見出し `2026 / 2027` による年単位の区切り表示。
* 表示列 `1列 / 2列 / 3列 / 4列` の切り替え。
* 表示モード `通常カード / コンパクト` の切り替え。
* フィルタ `すべて / HOME / AWAY / 2026 / 2027 / 未確定 / 枠線あり / 赤色枠 / 水色枠`。
* カードタップによる `未選択 / 赤色枠 / 水色枠` 状態の切り替え。
* LocalStorageによるカード状態、表示列、フィルタ条件、表示モードの保存。
* 作成者表記。
* 非公式ページとしての免責事項。
* 表示、絞り込み、保存状態をまとめる設定パネル。
* 操作方法を説明する使い方パネル。

## 想定する利用シーン

スクショ用表示とSNS投稿補助は、次のような利用シーンを想定します。

* 自分の参戦予定をまとめて投稿する。
* 気になる試合だけをまとめて投稿する。
* ホーム戦だけを投稿する。
* アウェイ遠征予定を投稿する。
* 2026年分だけを投稿する。
* 2027年分だけを投稿する。
* 未確定試合だけを確認・共有する。
* シーズン全体の日程を画像で共有する。

ただし、現時点では赤色枠・水色枠の意味をページ側では固定しません。利用者が赤色枠や水色枠にどのような意味を持たせるかは自由にし、投稿文や凡例でも決めつけない方針を維持します。

## スクショ用表示の基本方針

スクショ用表示は、通常閲覧画面とは別に、SNS共有しやすい見た目へ整える表示です。

基本方針は次のとおりです。

* 日程カード部分を主役にする。
* 設定ボタン、使い方ボタン、設定パネル、使い方パネルはスクショ用表示では非表示にする。
* 免責事項は長文のまま表示せず、必要最小限の注意書きにするか、スクショ外に逃がす案を検討する。
* 作成者表記は小さく残す。
* 現在のフィルタ結果を反映する。
* 赤色枠・水色枠の状態を反映する。
* 画像共有に適した余白、背景、密度を検討する。
* スマートフォンの縦長スクショを前提にする。
* 2列コンパクト表示と相性がよい設計にする。

スクショ用表示は、通常UIを完全に置き換えるものではなく、一時的に共有向けの見た目へ切り替える機能として扱います。

## SNS投稿補助の基本方針

SNS投稿補助は、スクショ用表示とは別に、投稿文を作りやすくする機能です。

基本方針は次のとおりです。

* X投稿向けの短い文面を生成できるようにする。
* フィルタ条件やカード状態を投稿文に反映する。
* 画像投稿だけでなくテキスト投稿もできるようにする。
* 投稿文はコピーできる形にする。
* 直接Xを開くリンクは将来候補として扱う。
* 投稿文には非公式であることを長く入れすぎない。
* 必要に応じてページURLやハッシュタグを入れる。
* アクセシビリティの観点から、画像だけで情報が完結しないようにする。

投稿文は利用者が編集できる前提にし、ページ側が利用者の意図を決めつけないようにします。

## 初期実装候補

初期実装では、実装範囲を小さく保ち、既存機能を壊さないことを優先します。

候補は次のとおりです。

* スクショ用表示に切り替えるボタン。
* スクショ用表示を終了するボタン。
* 投稿文を生成するボタン。
* 投稿文をコピーするボタン。
* 作成者表記。
* 簡易注意書き。

初期実装では、次の機能までは行いません。

* 画像ファイルの自動生成。
* 画像ダウンロード。
* Xへの自動投稿。
* 外部API連携。
* ログイン機能。

## 将来拡張候補

将来拡張候補は次のとおりです。

* Web Share APIによる共有。スマートフォンの共有シートを開き、投稿文やページURLを他アプリへ渡せるようにする案です。
* X投稿画面を開くリンク。生成した投稿文を使ってXの投稿画面を開く案ですが、自動投稿ではなく利用者が内容を確認して投稿する前提にします。
* 投稿文テンプレート選択。全体共有、HOME共有、AWAY共有、色枠状態共有など、用途別に文面を選べるようにします。
* ハッシュタグ設定。`#京都サンガ` などのハッシュタグを利用者が選択または編集できるようにします。
* スクショ用テーマ切替。背景色、余白、見出しデザインなどを共有先に合わせて切り替える案です。
* A4印刷 / PDF向け表示。SNSだけでなく紙やPDFで確認しやすい表示を用意します。
* 画像生成ライブラリによるPNG出力。将来的にブラウザ上で共有画像を生成する案ですが、初期実装では依存関係を追加しません。
* 選択試合だけの共有。赤色枠・水色枠や個別選択を使い、特定の試合だけをまとめて共有します。
* 観戦準備メモ込みの共有。移動、持ち物、集合時間などの個人メモを含める案ですが、公開JSONには含めず端末内情報として扱います。
* チケット販売日程との連携。将来、チケット発売日や販売URLを日程共有に添えられるようにします。
* カレンダー連携。選択した試合を個人カレンダーへ追加する機能と、共有表示を連動させる案です。

## 設定パネルとの関係

スクショ用表示・SNS投稿補助の入口は、設定パネルかフッター付近に置く方針です。日程表示欄の上部にボタンを増やしすぎないようにします。

現在の設定パネル内には「表示」「絞り込み」「保存状態」があるため、初期実装では「共有」または「スクショ・投稿」セクションを追加する案が自然です。スクショ用表示中は通常の設定パネルUIを隠す案も検討します。

構成案は次のとおりです。

```text
設定
├─ 表示
│  ├─ 表示列
│  └─ 表示モード
├─ 絞り込み
├─ 共有
│  ├─ スクショ用表示にする
│  ├─ 投稿文を作成
│  └─ 投稿文をコピー
└─ 保存状態
```

## フィルタリングとの関係

スクショ用表示は、現在のフィルタ結果を対象にします。

* HOMEフィルタならHOMEだけのスクショにする。
* AWAYフィルタならAWAYだけのスクショにする。
* 赤色枠フィルタなら赤色枠の試合だけを対象にする。
* 水色枠フィルタなら水色枠の試合だけを対象にする。
* 0件時はスクショ用表示にしても「該当する試合はありません」を表示する。
* 年見出しの表示・非表示制御は維持する。
* フィルタ条件を投稿文へ反映する。

フィルタを変えても、カード状態は削除しません。スクショ用表示は表示対象を絞るだけで、保存済み状態を変更しない方針です。

## パーソナライズ状態との関係

赤色枠・水色枠の意味はページ側で固定しません。投稿文でも「参戦予定」「気になる」と決めつけず、「赤色枠の試合」「水色枠の試合」のような中立表現にします。

将来、利用者がラベルを設定できるようになった場合は、そのラベルを投稿文へ反映する案を検討します。スクショ用表示では赤色枠・水色枠の凡例を小さく表示する案もありますが、その場合も意味を固定しません。

## 表示列・表示モードとの関係

スクショ用表示では、通常の表示列設定をそのまま使う案と、共有向けの列数へ一時的に切り替える案があります。

* 初期実装では、スクショ用表示中だけ一時的に2列・コンパクト相当の見た目にする。
* スマートフォンの縦長スクショで、より多くの試合が入る密度を優先する。
* 利用者の保存済み表示列・表示モードは変更しない。
* スクショ用表示を終了すると元の表示に戻す。
* LocalStorageの保存値を勝手に変更しない。

## 画面構成案

スクショ用表示の画面構成案は次のとおりです。

```text
[通常表示に戻るボタン]

2026
[試合カード][試合カード]
[試合カード][試合カード]

作成者：Kou from OSAKA
非公式の確認用日程表です。最新情報は公式情報をご確認ください。
```

方針は次のとおりです。

* 通常のフッター操作ボタンは非表示にする。
* 作成者表記は残す。
* 免責事項は短くする。
* 長い注記は必要に応じて表示する。
* 日程未確定や注記番号は消さない。

## 投稿文生成案

投稿文は、利用者がコピーして編集できる前提にします。

### 全体共有

```text
京都サンガF.C. 2026/27シーズン日程を確認中。
全38試合を表示しています。
#京都サンガ
```

### HOMEフィルタ

```text
京都サンガF.C. 2026/27シーズンのHOME戦を確認中。
19試合を表示しています。
#京都サンガ
```

### 赤色枠フィルタ

```text
京都サンガF.C. 2026/27シーズン日程。
赤色枠の試合を表示しています。
#京都サンガ
```

### AWAYフィルタ

```text
京都サンガF.C. 2026/27シーズンのAWAY戦を確認中。
遠征予定の整理に使っています。
#京都サンガ
```

注意点は次のとおりです。

* 赤色枠・水色枠の意味を固定しない。
* 投稿文は利用者が編集できる前提にする。
* 投稿文に長い免責事項は入れない。
* 必要ならページURLを含める案も検討する。

## アクセシビリティ上の注意

スクショ用表示とSNS投稿補助では、画像共有の見やすさだけでなく、テキスト情報へのアクセスも重視します。

* 画像投稿だけに依存しない。
* 投稿文をコピーできるようにする。
* スクショ用表示でもテキスト情報をHTML上に残す。
* 赤色枠・水色枠は色だけで伝えない。
* ボタンには分かりやすいラベルを付ける。
* `aria-live` で投稿文生成やコピー完了を伝える。
* スクショ用表示終了ボタンをキーボード操作できるようにする。
* Escキーでスクショ用表示を終了できるか検討する。
* `prefers-reduced-motion` に配慮する。
* X投稿時は画像の代替テキストや本文併記を推奨する。

## LocalStorage保存方針

既存キーは次のとおりです。

```text
sanga-schedule-button-states-v1
sanga-schedule-layout-v1
sanga-schedule-filter-settings-v1
sanga-schedule-display-mode-v1
```

スクショ用表示・SNS投稿補助では、初期実装では新しいLocalStorageキーを増やさない方針にします。

理由は次のとおりです。

* スクショ用表示は一時的な状態でよい。
* 投稿文は生成時点の状態から作る。
* 保存項目を増やすと保存内容削除の対象が増える。
* まずは実装を小さく保つ。

将来候補として、次のキーは検討可能です。

```text
sanga-schedule-share-settings-v1
```

保存候補は次のとおりです。

* 投稿文テンプレート。
* ハッシュタグ。
* スクショ用テーマ。
* 作成者表記の表示有無。
* ページURLを含めるか。

## 実装時の注意

実装時の注意は次のとおりです。

* 今回は設計のみで実装しない。
* 実装時は `public/data/matches.json` を変更しない。
* スクショ用表示はCSSクラス切替で実現できるか検討する。
* 既存の表示列・表示モードの保存値を勝手に変更しない。
* フィルタ処理、年見出し制御、カード状態を壊さない。
* パネル開閉、Escキー、フォーカス戻しを壊さない。
* スクショ用表示中に通常UIを隠す場合、戻し忘れがないようにする。
* View Transitions APIを使う場合は未対応環境にフォールバックする。
* `prefers-reduced-motion` を守る。
* HTML / CSS / JavaScriptを変更する実装時は、キャッシュクエリとページ下部バージョン更新を忘れない。

* 今回のPRで実装しない。

* ログイン機能を作らない。
* 認証情報を扱わない。

* 画像生成ライブラリを追加しない。
* 依存関係を追加しない。
* `public/data/matches.json` を変更しない。
* 赤色枠・水色枠の意味を固定しない。
