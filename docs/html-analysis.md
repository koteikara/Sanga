# `public/sanga202627season.html` 分析メモ

## 1. 現在のHTML構造の概要

対象ファイルは、京都サンガF.C. 2026/27シーズンの年間日程を、スマートフォン幅を主軸にしたカード型UIで表示する単一HTMLです。

大まかな構成は次の通りです。

```text
html
├─ head
│  ├─ meta charset / viewport
│  ├─ title
│  └─ style（全CSSをインライン定義）
└─ body
   └─ main.phone
      ├─ header
      │  ├─ .league（CSSで非表示）
      │  └─ h1
      ├─ section#y2026
      │  ├─ .year
      │  └─ .grid
      │     └─ button.match × 20
      ├─ section#y2027
      │  ├─ .year
      │  └─ .grid
      │     └─ button.match × 18
      ├─ .footer-tools
      │  ├─ .footer-actions
      │  │  ├─ 使い方ボタン
      │  │  └─ 表示列切替ボタン（2列 / 3列 / 4列）
      │  └─ .legend（注記 ※1〜※7）
      ├─ .disclaimer（免責事項）
      ├─ .help-overlay
      └─ .help-panel（使い方ダイアログ）
script（全JavaScriptをインライン定義）
```

各試合は `button.match` として実装され、内部に次の情報を持っています。

* `data-id`: `sec01` 〜 `sec38` の試合識別子
* `home` / `away`: ホーム・アウェイの表示制御クラス
* `logo-*`: 対戦相手ロゴ背景の表示制御クラス
* `.sec`: 節情報
* `.team`: 対戦相手の短縮表示名
* `.place`: 会場表示
* `.date`: 日付表示
* `.note`: 注記番号（必要な試合のみ）

## 2. HTML内に直接書かれている日程データ・表示データの種類

現在のHTMLには、表示に必要なデータが各 `button.match` の中に直接書かれています。主な種類は次の通りです。

| 種類 | 現在の持ち方 | 例 |
| --- | --- | --- |
| 試合ID | `data-id` | `sec01`, `sec38` |
| 年 | 年別 `section` と `.year` | `2026`, `2027` |
| 節 | `.sec` | `第1節` |
| ホーム/アウェイ | `home` / `away` クラス、縦書き風の `HOME` / `AWAY` 表示 | `class="match home ..."` |
| 対戦相手 | `.team` と `logo-*` クラス | `川崎F`, `logo-frontale` |
| 会場 | `.place` | `サンガスタジアム`, `未定` |
| 日付 | `.date` 内の月日・曜日 | `9.2 WED`, `12.19 SAT` |
| 候補日 | `.date .range` | `8.8 SAT` / `8.9 SUN` |
| 未確定表示 | `.tentative-date` クラス | 淡い黄色の日付枠 |
| 注記番号 | `.note` | `※1` 〜 `※7` |
| 注記本文 | `.legend` | 開催候補日、ACL Elite・ルヴァンカップ関連の注記 |
| 免責事項 | `.disclaimer` | 非公式・正確性保証なし等 |
| ヘルプ文言 | `.help-panel` | タップ状態、表示列、LocalStorageの説明 |
| 表示制御文言 | ボタンラベル | `使い方`, `2列`, `3列`, `4列` |

また、対戦相手ロゴはCSS内のBase64画像スプライトと `logo-*` クラス別の `background-position` によって表示されています。これは日程データというより表示アセットですが、JSON化時には「対戦相手コード」や「ロゴクラス」をどこで管理するかを決める必要があります。

## 3. スプレッドシート管理に移せそうな項目

`docs/data-schema.md` の定義に沿って、次の項目はスプレッドシート管理に移しやすいです。

| 現在のHTML上の情報 | スプレッドシート列候補 | JSON内部キー候補 | 補足 |
| --- | --- | --- | --- |
| `sec01` など | ID | `id` | 既存の `secNN` を残すか、`2026-j1-01` 形式へ寄せるか要検討 |
| 2026/2027 | シーズン / 試合日 | `season`, `match_date` | 表示年は日付から生成可能 |
| 第1節など | 節・回戦 | `round` | そのまま移行可能 |
| HOME / AWAY | ホーム/アウェイ | `home_away` | `H` / `A` へ変換可能 |
| 長崎、川崎Fなど | 対戦相手 | `opponent` | 表示名と正式名を分ける余地あり |
| `logo-vvaren` など | 対戦相手コード / 表示用クラス | `opponent_code` など | 現行スキーマにはないため追加検討 |
| サンガスタジアム / 未定 | 会場 | `venue` | 将来的に会場マスタ化可能 |
| 月日・曜日 | 試合日 | `match_date` | 候補日がある場合は単一日付だけでは不足 |
| 未定 | 状態 / 備考 | `status`, `note` | 未確定・確認中の扱いを統一したい |
| 候補日レンジ | 状態 / 備考 / 追加キー | `status`, `note`, `date_candidates` など | 現行スキーマ外。JSON化前に設計が必要 |
| `※1` 〜 `※7` | 備考 / 注記ID | `note` / `note_id` | 複数試合で再利用するなら注記マスタも有効 |
| 注記本文 | 備考または注記シート | `note` / `notes` | 長文のため試合行と分ける案もある |
| 免責事項 | サイト固定文言 | `meta.disclaimer` など | 日程データとは分離が望ましい |

特に候補日レンジ、曜日表示、ロゴクラス、注記本文は、現行の `docs/data-schema.md` だけでは表現しきれない可能性があります。小さく始めるなら、まずは `note` と `status` で保持し、後から `date_candidates` や `opponent_code` を追加するのが安全です。

## 4. CSSとして分離できそうな部分

現在はすべてのCSSが `<style>` 内にあります。次のまとまりで `public/assets/style.css` へ分離できます。

1. **基本レイアウト**
   * `:root` の色変数
   * `html`, `body`, `.phone`
   * `header`, `h1`, `.year`, `.grid`

2. **試合カードUI**
   * `.match`, `.match-inner`
   * `.home`, `.away`
   * `.ha`, `.meta`, `.sec`, `.team`, `.place`, `.date`, `.note`
   * `data-state="1"`, `data-state="2"` の選択状態

3. **未確定日程・曜日表示**
   * `.tentative-date`
   * `.sub.sat`, `.sub.sun`
   * 候補日レンジ `.date .range`

4. **レスポンシブ・コンパクト表示**
   * `@media (min-width:390px)`
   * `@media (min-width:700px)`
   * `.layout-3`, `.layout-4` 関連

5. **ヘルプUI**
   * `.footer-tools`, `.footer-actions`
   * `.help-button`, `.help-overlay`, `.help-panel`, `.help-close`
   * `prefers-reduced-motion` 対応

6. **対戦相手ロゴ表示**
   * `.meta::after`
   * `logo-*` ごとの `background-position`
   * 現状はBase64スプライトが非常に大きいため、CSS分離時には可読性・差分管理の観点で外部画像ファイル化も検討できます。

注意点として、CSSには後段で同じセレクタを上書きしている箇所が多くあります。単純に並び替えると表示崩れが起きるため、初回の分離では順序を変えずにそのまま外部CSSへ移すのが安全です。

## 5. JavaScriptとして分離できそうな部分

現在のJavaScriptは `<script>` 内の即時関数にまとまっています。`public/assets/app.js` へ分離しやすい機能は次の通りです。

1. **試合カードのタップ状態管理**
   * `.match` を取得
   * `data-state` を `0 → 1 → 2 → 0` の順に切り替え
   * `aria-pressed` を更新
   * LocalStorageへ保存

2. **使い方パネルの開閉**
   * `使い方` ボタンで開く
   * `閉じる` ボタン、オーバーレイクリック、Escapeキーで閉じる
   * `hidden`, `.is-open`, `aria-expanded` を更新
   * 閉じた後に起点ボタンへフォーカスを戻す

3. **表示列切替**
   * `2列` / `3列` / `4列` ボタンの状態管理
   * `.phone` に `.layout-3` / `.layout-4` を付与
   * `aria-pressed` を更新
   * LocalStorageへ保存

4. **保存内容削除**
   * 試合カード状態と表示列設定をLocalStorageから削除
   * DOM上の状態を初期化
   * `aria-live` 領域に削除完了メッセージを表示

分離時は、HTMLの読み込み後にDOMを参照する必要があります。現状は `</main>` 後に `<script>` があるため動作しています。外部JS化する場合は、同じ位置で `<script src="assets/app.js"></script>` を置くか、`defer` を付けるのが安全です。

## 6. LocalStorageを使っている機能の整理

現在LocalStorageで保存している値は2種類です。

| キー | 用途 | 保存内容 | 復元タイミング |
| --- | --- | --- | --- |
| `sanga-schedule-button-states-v1` | 試合カードごとのタップ状態 | `{ "sec01": 1, "sec02": 2 }` のようなオブジェクト | ページ読み込み時に各 `.match` へ `data-state` と `aria-pressed` を反映 |
| `sanga-schedule-layout-v1` | 表示列設定 | `'2'`, `'3'`, `'4'` | ページ読み込み時に `.phone` のレイアウトクラスと各ボタンの `aria-pressed` を反映 |

試合カードの状態は次の3段階です。

| `data-state` | 見た目 | 意味 |
| --- | --- | --- |
| `0` | 通常枠 | 未選択 |
| `1` | 赤色枠 | 1回タップ状態 |
| `2` | 水色枠 | 2回タップ状態 |

ヘルプ内の「このページの保存内容を削除」ボタンは、両方のLocalStorageキーを削除し、画面上のカード状態と表示列を初期状態へ戻します。

JSON化時の注意として、LocalStorageのキーが `secNN` に依存しています。今後IDを `2026-j1-01` 形式へ変える場合、既存ユーザーの保存状態が引き継げなくなる可能性があります。互換性を重視するなら、移行期間は旧IDと新IDの対応表を用意する案があります。

## 7. アクセシビリティ上、維持・改善した方がよい点

### 維持した方がよい点

* `html lang="ja"` が設定されています。
* 年別セクションに `aria-labelledby` が使われています。
* 試合カードが `button` なので、キーボード操作とフォーカス対象として扱えます。
* タップ状態に `aria-pressed` を付与しています。
* 表示列切替は `role="group"` と `aria-pressed` を使っています。
* 使い方パネルは `role="dialog"`, `aria-modal="true"`, `aria-labelledby` を持っています。
* `Escape` キーで使い方パネルを閉じられます。
* ヘルプ閉鎖後に起点ボタンへフォーカスを戻しています。
* `prefers-reduced-motion` に対応しています。
* 土曜・日曜の色を青・赤で分けつつ、文字ラベルも表示されています。
* 保存削除メッセージに `aria-live="polite"` が使われています。

### 改善した方がよい点

* 試合カードのアクセシブル名が、内部テキストの連結に依存しています。JSON化やテンプレート化の際に `aria-label="第1節 アウェイ 長崎 8月8日または8月9日 会場未定"` のような明示ラベルを生成すると、スクリーンリーダーで理解しやすくなります。
* `HOME` / `AWAY` が1文字ずつ分割された装飾テキストとして入っています。読み上げが不自然になる可能性があるため、装飾部分を `aria-hidden="true"` にし、カード側の `aria-label` で意味を補う案があります。
* `.help-panel` は開いた時に閉じるボタンへフォーカスしますが、ダイアログ内のフォーカストラップはありません。将来的に操作要素が増える場合は検討対象です。
* ヘルプ内の `<ul>` 直下に `button` と `p` が置かれているため、HTML構造としては改善余地があります。リスト項目に含めるか、`</ul>` の外へ出す方が自然です。
* 試合一覧は多数のボタンが連続するため、セクションごとに「2026年の試合一覧」などの見出し・説明を明確に維持するとよいです。
* 色付き枠の意味はヘルプで説明されていますが、カード自体の状態名は `aria-pressed=true/false` だけでは赤色枠・水色枠の違いを伝えにくいです。将来的には `aria-label` や視覚的に隠したテキストで状態名を含めるとよいです。

## 8. JSON化するときの注意点

1. **日付候補の表現**
   * 現在は単一日付、候補日レンジ、未定、注記付き日程が混在しています。
   * `match_date` だけでなく、候補日を配列で持つ `date_candidates` のような項目が必要になる可能性があります。

2. **表示用日付とデータ用日付を分ける**
   * HTMLでは `8.8` や `SAT` のような表示文字列を直接持っています。
   * JSONでは `YYYY-MM-DD` を正本にし、表示時に `8.8 SAT` を生成する方が安全です。

3. **未確定状態の管理**
   * `.tentative-date` は見た目のクラスです。
   * JSONでは `status: "tentative"` などの意味データとして持ち、表示時にクラスへ変換するのが望ましいです。

4. **注記の管理**
   * `※1` 〜 `※7` は複数の意味を持つ長文注記です。
   * 試合データに全文を入れると重複や更新漏れが起きやすいため、`note_id` と注記一覧を分ける案があります。

5. **対戦相手名・ロゴ・正式名の扱い**
   * 現在の `.team` は短縮名です。
   * スプレッドシート上は正式名、表示用短縮名、ロゴ用コードを分けると拡張しやすいです。

6. **既存LocalStorageとの互換性**
   * 保存状態は `data-id` に依存しています。
   * ID体系を変える場合は、旧IDの保存状態が消えることを許容するか、移行処理を入れるか判断が必要です。

7. **HTML生成時のアクセシブル名**
   * JSONからカードを生成する場合、視覚表示だけでなく `aria-label` や `aria-pressed` も同時に生成・更新する必要があります。

8. **データ欠損時の表示**
   * 会場未定、日付未定、対戦相手未定などを想定し、空欄でもカードが破綻しないデフォルト表示を決める必要があります。

9. **公式情報の出典**
   * `source_url` と `source_checked_at` をJSONに含められる設計にしておくと、後から確認・更新しやすくなります。

## 9. 次に実施する改修案（小さな単位）

1. **CSSの外部ファイル化だけを行う**
   * `<style>` の内容を順序を変えずに `public/assets/style.css` へ移す。
   * HTML側は `<link rel="stylesheet" href="assets/style.css">` に置き換える。
   * 表示差分が出やすいため、まずは内容の整理や削除をしない。

2. **JavaScriptの外部ファイル化だけを行う**
   * `<script>` の内容を `public/assets/app.js` へ移す。
   * HTML側は `defer` または末尾読み込みで参照する。
   * LocalStorageのキー名と動作は変えない。

3. **試合データのサンプルJSONを作成する**
   * まずは `public/data/matches.sample.json` に数試合分だけを手入力する。
   * 候補日、未定、注記付きの代表例を含める。
   * 既存HTMLの表示はまだJSON読み込みへ切り替えない。

4. **HTMLカード構造のテンプレート方針を文書化する**
   * JSONの各項目が、カード内のどの要素・クラス・ARIA属性へ反映されるかを整理する。
   * `docs/data-schema.md` に `date_candidates`, `opponent_code`, `short_opponent_name`, `note_id` などを追加するか検討する。

5. **アクセシブル名の改善を小さく試す**
   * 1〜2件のカードだけを対象にするのではなく、生成ルールを決めたうえで一括反映する方が安全。
   * 実装前に、`aria-label` の文言ルールを文書化する。
