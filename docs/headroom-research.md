# Codex作業時のheadroom導入可能性調査

## この調査の目的

この調査は、Codex作業時に毎回読み込む長い文脈や検証ログを、`headroom` で安全に圧縮できるかを整理するものです。

このリポジトリでは、Codexへ作業を依頼する際に、`AGENTS.md`、`README.md`、各種docs、HTML / CSS / JavaScript、GitHub Pages確認項目、PR作成ルール、変更禁止ファイル、LocalStorageキー、アクセシビリティ要件、表記統一ルールなどを渡すことがあります。そのため、入力トークンが大きくなりやすい状態です。

ただし、今回の調査では `headroom` 本体の導入、インストール、設定変更は行いません。導入可能性、期待効果、懸念点、小規模検証案をドキュメントとして整理することに留めます。

## headroomの概要

確認元: <https://github.com/chopratejas/headroom>

### READMEで確認できた情報

`headroom` は、AIエージェントやLLMが読むコンテキストを、LLMへ届く前に圧縮することを目的にしたツールです。READMEでは、tool output、ログ、RAGチャンク、ファイル、会話履歴などを圧縮対象として説明されています。

README上では、次のような使い方が提示されています。

* PythonまたはTypeScriptから使うライブラリ
* OpenAI互換クライアントなどの前段に置くプロキシ
* `claude`、`codex`、`cursor`、`aider`、`copilot` などのエージェントを包むラップ実行
* MCPサーバーとしての利用
* 複数エージェント間での共有メモリ
* 失敗セッションから `AGENTS.md` などへ改善指示を書き出す学習系コマンド

READMEでは、JSON、コード、自然言語、ログのような異なる種類の入力を判定し、内容に応じて圧縮方式を切り替える構成が説明されています。また、元の内容をローカルにキャッシュし、必要に応じて取得できる可逆的な圧縮の考え方も示されています。

ユースケースとしては、AIコーディングエージェントの利用、コード検索結果の圧縮、SREインシデント調査ログの圧縮、GitHub Issueトリアージ、コードベース探索などが挙げられています。

環境面では、Pythonパッケージ、Node / TypeScriptパッケージ、Dockerイメージの利用が提示されています。READMEでは Python 3.10 以上が必要とされています。Node / TypeScriptからの利用も示されていますが、このリポジトリで実際にどの機能をどこまで使えるかは未確認です。

### 未確認の情報

次の点は、READMEだけではこのプロジェクトでの利用可否を断定できません。

* Codexクラウド実行環境に `headroom` を直接導入できるか。
* この会話のChatGPT入力トークンを直接減らせるか。
* `headroom wrap codex` が、このリポジトリで使っているCodex実行形態と同じ前提で動くか。
* GitHub Actions内で利用した場合に、どのログをどのタイミングで圧縮できるか。
* 圧縮結果が日本語の細かい表記統一やPRルールを十分に保持できるか。
* セキュリティ、ライセンス、メンテナンス状況を本番運用に耐える水準で確認できているか。

### 推測として考えられること

`headroom` は、Codexや周辺CLIが読む長いログ、差分、検索結果、ドキュメントの要約には効果がある可能性があります。一方で、PR本文の見出し、変更禁止ファイル、LocalStorageキー、表記統一ルールのように、1文字単位で正確性が必要な情報を圧縮して渡す用途には慎重な検証が必要です。

## 期待できる効果

このプロジェクトで期待できそうな効果は次のとおりです。

* Codexが読むファイル量の削減
* GitHub差分確認時のトークン削減
* 長いログや検証結果の圧縮
* PR履歴や作業文脈の圧縮
* 毎回の周辺情報読み込み量の削減
* コード検索結果やドキュメント検索結果の要約
* 長いGitHub Actionsログからエラー箇所を探す負荷の軽減

ただし、次の制限を明確にしておく必要があります。

* ChatGPTのこの会話自体のトークン消費を直接削減するものではありません。
* Codexクラウド環境に直接導入できるかは要確認です。
* 圧縮により重要な細部が欠落するリスクがあります。
* 圧縮前の原文を確認する手順を残さないと、誤った要約を前提に作業する可能性があります。
* 認証情報やSecretsを含む情報は、圧縮対象にする前に扱いを慎重に決める必要があります。

## このプロジェクトで効果がありそうな場面

| 場面 | 効果の見込み | 理由 |
| --- | --- | --- |
| Codexが複数ドキュメントを読むとき | ありそう | `AGENTS.md`、README、運用docsの共通部分を短く把握できる可能性があるため。 |
| HTML / CSS / JS を同時に読むとき | 慎重にありそう | 画面構造の把握には有効そうだが、セレクタ名、LocalStorageキー、イベント処理の細部は原文確認が必要なため。 |
| GitHub PR差分を読むとき | ありそう | 変更概要、影響範囲、注意点の把握に使える可能性があるため。 |
| GitHub Actionsログを読むとき | ありそう | 長いログから失敗箇所や警告だけを抽出する用途に向きそうなため。 |
| 検証コマンドの長い出力を読むとき | ありそう | エラー行、失敗した検証項目、件数差分の把握に使える可能性があるため。 |
| 過去のPR文脈を参照するとき | ありそう | 直近PRの目的、変更範囲、未対応事項を要約できる可能性があるため。 |
| `docs/operation-flow.md` や `docs/filtering.md` など長いドキュメントを参照するとき | 慎重にありそう | 大枠理解には向くが、手順、禁止事項、表記統一、確認項目は原文参照が必要なため。 |

## Codex作業との相性

このプロジェクトでは、Codex作業時の制約が細かく、圧縮対象に向く情報と向かない情報を分ける必要があります。

| 情報 | 圧縮対象に向くか | 理由 |
| --- | --- | --- |
| 長いログ | 向く | エラー箇所、警告、失敗したコマンドの要約が有効なため。 |
| GitHub Actionsログ | 向く | 成功部分を省き、失敗ステップや関連ログを抽出しやすいため。 |
| PR差分概要 | 向く可能性あり | 変更点把握には有効だが、最終確認は原文差分が必要なため。 |
| コード検索結果 | 向く可能性あり | 関連箇所の候補を絞る用途に使えるため。 |
| `docs/operation-flow.md` の全体像 | 慎重 | 手順理解には有効だが、確認項目やデプロイ条件の欠落は避ける必要があるため。 |
| PRルール | 向かない | PRタイトル、PR本文見出し、禁止見出しなどを正確に保持する必要があるため。 |
| 表記統一ルール | 向かない | 「赤色枠」「水色枠」など、1文字単位の表記差が問題になるため。 |
| 変更禁止ファイル一覧 | 向かない | `public/data/matches.json` などの禁止対象が1つでも落ちると事故につながるため。 |
| LocalStorageキー | 向かない | キー名が変わると既存利用者の保存状態を壊す可能性があるため。 |
| アクセシビリティ要件 | 慎重 | 要約は可能だが、見出し構造、キーボード操作、色だけに依存しない設計などが欠落すると品質低下につながるため。 |
| 認証情報、Secrets、`.env` | 向かない | 圧縮・送信以前に、扱いを最小化し、リポジトリやログへ出さないことが重要なため。 |

特に、次のような制約は圧縮せず、原文または短い固定テンプレートとして維持する方が安全です。

* PR本文は日本語見出しにする。
* 英語見出しを使わない。
* 変更禁止ファイルを守る。
* LocalStorageキーを変更しない。
* 赤色枠 / 水色枠の表記を守る。
* アクセシビリティ要件を守る。
* GitHub Pages確認項目をPR本文に入れる。
* 認証情報を含めない。

## 懸念点

`headroom` の導入には、次の懸念があります。

* 重要な指示が圧縮で欠落する可能性があります。
* 表記統一が崩れる可能性があります。
* アクセシビリティ要件が省略される可能性があります。
* 実装禁止事項が落ちる可能性があります。
* 圧縮結果の検証が必要です。
* 導入により作業フローが複雑になります。
* Codexクラウド環境で使えない可能性があります。
* ライブラリのメンテナンス状況・ライセンス確認が必要です。
* セキュリティ面の確認が必要です。
* GitHubやSecrets、認証情報に触れる情報を圧縮・送信する場合の扱いに注意が必要です。
* ローカルにキャッシュされる原文の保存場所、TTL、削除方法を確認する必要があります。
* GitHub Actionsで使う場合、ログの公開範囲や一時Public運用時のリスクを確認する必要があります。
* `headroom learn` のように `AGENTS.md` へ書き出す機能は便利そうですが、このリポジトリでは人間がレビューした小さな変更だけに限定する必要があります。

## 導入しない場合の代替策

`headroom` を導入しなくても、トークン消費を抑える運用改善は可能です。

* 共通指示を `docs/codex-workflow.md` に集約する。
* 毎回のCodex指示書を短縮する。
* PR作成ルールを `AGENTS.md` に寄せる。
* 変更禁止事項をテンプレート化する。
* 対象ファイルだけを明示する。
* 作業種別ごとの短い指示テンプレートを用意する。
* 長い検証項目をdocs側に寄せる。
* 「今回はこのセクションだけ参照」と指定する。
* PR本文テンプレートを日本語見出しで固定する。
* 変更禁止ファイルのチェックコマンドを用意する。
* GitHub Pages確認項目と本番デプロイ確認項目を、短いチェックリストとして別ファイル化する。

現時点では、`headroom` 導入より先に、Codex指示書の共通化・短縮化の方が安全で効果が出やすい可能性があります。特に、このプロジェクトでは正確な日本語表記、禁止事項、公開運用ルールが重要なため、まずは人間が確認しやすい短いテンプレートを整備する方が安全です。

## 小規模検証案

導入を判断する前に、認証情報やSecretsを含まないサンプルだけで小規模に検証します。

1. `headroom` をローカル環境で試す。
2. 長いGitHub Actionsログを圧縮して意味が保たれるか確認する。
3. `docs/filtering.md` を圧縮して重要要件が残るか確認する。
4. Codex指示書を圧縮してPRルールが欠落しないか確認する。
5. 圧縮前後でCodexの修正品質を比較する。
6. 認証情報やSecretsを含まないサンプルだけで試す。
7. `docs/operation-flow.md` の手順要約で、GitHub Pages確認、本番デプロイ条件、Secrets禁止が残るか確認する。
8. `public/assets/app.js` のようなコードを圧縮した場合に、LocalStorageキーやイベント処理の重要点が残るか確認する。

検証時の確認項目は次のとおりです。

* トークン削減率
* 重要要件の保持率
* 表記統一ルールが残るか
* 変更禁止ファイルが残るか
* アクセシビリティ要件が残るか
* Codexの出力品質
* 作業時間増減
* 導入手順の複雑さ
* 圧縮前原文へ戻れるか
* ローカルキャッシュの保存場所と削除方法
* 日本語ドキュメントでの要約品質
* Secretsや認証情報を含まない運用にできるか

## 現時点の結論

`headroom` は、Codex作業時のログ、差分、長文コンテキスト圧縮には有望です。READMEでは、Codexを含むエージェントラップ、プロキシ、MCP、ライブラリ利用が提示されており、ログや検索結果のような長い情報を短く扱う用途には効果が期待できます。

ただし、このプロジェクトでは、PR本文の日本語見出し、変更禁止ファイル、LocalStorageキー、赤色枠 / 水色枠の表記、アクセシビリティ要件、GitHub Pages確認項目、認証情報を含めない運用など、細かな制約や表記統一が重要です。そのため、圧縮で重要な条件が欠落するリスクを考えると、現時点で即導入はしません。

まずはこの調査ドキュメントをもとに、認証情報を含まないサンプルで小規模検証します。並行して、Codex指示書の共通化・短縮化も進めます。実際に導入する場合も、PRルール、変更禁止ファイル、認証情報禁止、アクセシビリティ要件のような重要制約は、圧縮済み要約だけに頼らず、原文または固定テンプレートとして保持します。
