# フィルタリング機能の設計

## この設計の目的

年間スケジュールページに、目的の試合だけを探しやすくするフィルタリング機能を追加するための設計を整理します。

年間スケジュールページは、スマートフォンで京都サンガF.C.の試合予定を確認し、閲覧者ごとのカード状態をLocalStorageに保存し、SNS投稿やスクショ用表示へつなげることを主な目的とします。今後、試合数が増えたり、カップ戦・ACL戦が追加されたり、利用者ごとの色枠状態やラベルが増えたりすると、必要な試合だけを絞り込めることが重要になります。

この文書では、実装前に次の点を整理します。

* フィルタ対象にする情報
* 初期実装で扱うフィルタ
* 将来拡張で扱うフィルタ
* パーソナライズ状態との関係
* UI方針
* LocalStorage保存方針
* 共有URLとの関係
* アクセシビリティ上の注意

今回の設計はドキュメント整理であり、HTML、CSS、JavaScript、JSONの変更を前提にしません。

## フィルタリングの基本方針

* スマートフォンで目的の試合を素早く探せることを重視します。
* 初期表示では全試合を表示します。
* フィルタを選ぶと、該当する試合だけを表示します。
* フィルタ解除で全試合表示に戻せるようにします。
* 表示されないカードのLocalStorage状態は削除しません。
* フィルタは閲覧者ごとの表示補助であり、スプレッドシートやJSONの正本データを変更しません。
* フィルタ条件は、必要に応じてLocalStorageに保存します。
* 将来的に共有URLでフィルタ条件を指定できる余地を残します。
* フィルタ結果は、SNS投稿、スクショ用表示、カレンダー連携などの対象選択にも使えるようにします。

## フィルタ対象

フィルタ対象は、全閲覧者に共通する「データ由来フィルタ」と、閲覧者ごとに変わる「LocalStorage由来フィルタ」に分けます。

### データ由来フィルタ

`public/data/matches.json` や、将来的なGoogleスプレッドシート由来の項目です。試合そのものの属性であり、閲覧者ごとに変わらない情報を対象にします。

候補は次のとおりです。

| フィルタ対象 | 対応するJSONキー候補 | 用途 |
| --- | --- | --- |
| 大会 | `competition` / `competition_label` | J1、ルヴァンカップ、天皇杯、ACL、親善試合などで絞り込む。 |
| 開催年 | `match_date` / `date_candidates` | 2026年、2027年など、年またぎシーズン内の試合を絞り込む。 |
| HOME / AWAY | `home_away` / `home_away_label` | ホーム、アウェイ、中立地、未定を絞り込む。 |
| 試合状態 | `match_status` | 試合前、試合中、試合終了、延期、中止などで絞り込む。 |
| 結果 | `result` | 勝、分、敗などで絞り込む。 |
| 表示フラグ | `is_visible` | 公開画面に表示する試合かどうかを制御する。 |
| 対戦相手 | `opponent` / `opponent_code` | 特定の対戦相手を絞り込む。 |
| 会場 | `venue` | サンガスタジアム、アウェイ会場などで絞り込む。 |
| 日程確定状態 | `status` / `date_candidates` | 確定、未確定、候補日ありなどで絞り込む。 |

### LocalStorage由来フィルタ

閲覧者ごとの状態です。スプレッドシートや公開JSONには保存せず、端末内のLocalStorageに保存された個人状態を対象にします。

候補は次のとおりです。

| フィルタ対象 | 現時点の扱い | 備考 |
| --- | --- | --- |
| 未選択 | 実装済みカード状態 `0` | 枠線なしの通常状態。 |
| 赤枠 | 実装済みカード状態 `1` | 意味は固定せず、利用者が自由に決める色枠として扱う。 |
| 水色枠 | 実装済みカード状態 `2` | 意味は固定せず、利用者が自由に決める色枠として扱う。 |
| 将来追加される色枠 | 将来機能 | 状態値や表示色を増やす場合に検討する。 |
| 利用者が設定したラベル | 将来機能 | 「参戦予定」などの意味をページ側で固定せず、利用者設定として扱う。 |
| メモあり | 将来機能 | 個人メモ機能が追加された場合に検討する。 |
| チケット購入済み | 将来機能 | 閲覧者個人の準備状態としてLocalStorageで扱う。 |

現時点で実装済みなのは、カード状態 `0 / 1 / 2` のみです。利用者ラベル、メモ、チケット購入済みなどは将来機能として扱います。

## 初期実装で扱うフィルタ

初期実装では、実装しやすく、スマートフォンで効果が大きいものに絞ります。

推奨する初期フィルタは次のとおりです。

```text
すべて
HOME
AWAY
2026
2027
未確定日程
枠線あり
赤枠
水色枠
```

各フィルタの方針は次のとおりです。

| フィルタ | 判定方針 |
| --- | --- |
| すべて | フィルタ解除として扱い、全試合を表示する。 |
| HOME | `home_away` が `H` の試合を対象にする。 |
| AWAY | `home_away` が `A` の試合を対象にする。 |
| 2026 | 開催日、候補日、節番号から判定する既存ロジックと整合させる。 |
| 2027 | 開催日、候補日、節番号から判定する既存ロジックと整合させる。 |
| 未確定日程 | `status: tentative` または候補日ありの試合を対象にする。 |
| 枠線あり | LocalStorage状態が `1` または `2` のカードを対象にする。 |
| 赤枠 | LocalStorage状態 `1` のカードを対象にする。 |
| 水色枠 | LocalStorage状態 `2` のカードを対象にする。 |

赤枠・水色枠の意味は固定しません。たとえば `赤枠 = 参戦予定` のような表現は避け、利用者が自分で意味を決める色枠として扱います。

## 将来拡張で扱うフィルタ

将来拡張では、試合数の増加、戦績更新、個人設定の拡張、SNS投稿やスクショ用表示との連携を見据えて、次のフィルタを検討します。

| フィルタ | 方針 |
| --- | --- |
| 大会別 | カップ戦・ACL戦を追加した場合は、大会別フィルタが重要になる。 |
| 試合状態別 | 試合前、試合中、試合終了、延期、中止などで絞り込む。 |
| 結果別 | 戦績更新後は、勝・分・敗の結果フィルタが使える。 |
| 対戦相手別 | 特定クラブとの試合を探しやすくする。 |
| 会場別 | ホームスタジアム、遠征先、未定会場などで絞り込む。 |
| 利用者ラベル別 | 利用者ラベル設定が実装されたら、赤枠・水色枠ではなくラベル名でのフィルタも検討する。 |
| メモあり | 追加メモ機能が実装されたら、メモありフィルタを検討する。 |
| チケット購入済み | チケット購入状態をLocalStorageで持つ場合に検討する。 |
| カレンダー登録対象 | カレンダー連携で登録したい試合の選択に使う。 |
| スクショ対象 | SNS投稿やスクショ用表示で、フィルタ結果をそのまま共有対象にできるようにする。 |

SNS投稿やスクショ用表示では、フィルタ結果をそのまま共有対象にできるようにします。ただし、共有されるべき内容と、閲覧者の端末内だけに保存される個人状態は分けて扱います。

## パーソナライズ状態との関係

`docs/personalization.md` の方針に合わせ、色枠の意味はページ側で固定しません。

* フィルタでは、まずは `赤枠`、`水色枠` のように見た目上の状態として扱います。
* `赤枠 = 参戦予定`、`水色枠 = 視聴予定` のような固定表現は避けます。
* 将来的に利用者ラベルが設定された場合は、ラベル名でフィルタ表示できるようにします。
* LocalStorage状態は、フィルタで非表示になっても保持します。
* 保存内容削除ボタンでは、フィルタ条件も削除対象にするか検討します。
* フィルタ条件とカード状態は別のLocalStorageキーに分ける方針がよいです。
* 利用者ごとの状態やラベルは、スプレッドシートや公開JSONに保存しません。

既存のカード状態保存キーは変更しません。

```text
sanga-schedule-button-states-v1
```

将来的なフィルタ条件保存キー案は次のとおりです。

```text
sanga-schedule-filter-settings-v1
```

保存イメージは次のとおりです。

```json
{
  "activeFilters": {
    "homeAway": "H",
    "year": "2026",
    "personalState": "1"
  }
}
```

## UI方針

スマートフォン向けUIとして、フィルタUIはコンパクトにします。初期画面がごちゃつかないよう、表示数や配置には注意します。

検討するUI方針は次のとおりです。

* フィルタUIはコンパクトにします。
* 常時大きく表示しすぎないようにします。
* フッター付近または日程上部に「絞り込み」ボタンを置く案を検討します。
* ボタンを押すとフィルタパネルを開く案を検討します。
* よく使うフィルタだけはチップボタンとして表示する案を検討します。
* 選択中のフィルタが分かるようにします。
* フィルタ解除ボタンを用意します。
* 表示件数を表示する案を検討します。
* フィルタ結果が0件の場合は、分かりやすいメッセージを表示します。

UI候補は次のとおりです。

```text
絞り込み
すべて
HOME
AWAY
2026
2027
未確定
枠線あり
赤枠
水色枠
解除
```

初期実装では、すべての候補を常時表示するのではなく、利用頻度が高いものだけをチップ表示し、詳細条件はパネル内に入れる構成も検討します。

## LocalStorage保存方針

フィルタ条件は、必要に応じてLocalStorageへ保存します。ただし、カード状態とは別キーで保存します。

* 既存のカード状態保存キー `sanga-schedule-button-states-v1` は変更しません。
* フィルタ条件用には `sanga-schedule-filter-settings-v1` のような別キーを検討します。
* フィルタ条件を保存する場合でも、スプレッドシートや公開JSONは変更しません。
* フィルタで非表示になったカードの状態は削除しません。
* フィルタ条件が壊れている、または未知の値を含む場合は、安全側に倒して全試合表示へ戻せるようにします。
* 保存内容削除ボタンでカード状態だけを消すのか、フィルタ条件も消すのかはUI文言とあわせて検討します。
* 個人ラベル、メモ、チケット購入済みなどが追加された場合も、公開JSONではなくLocalStorage側に保存します。

## 共有URLとの関係

将来的に共有URL機能を追加する場合に備えて、フィルタ条件をURLクエリで表現できる余地を残します。

例:

```text
?homeAway=H
?year=2026
?status=tentative
?state=1
?competition=J1
```

方針は次のとおりです。

* URLで共有するのはフィルタ条件です。
* 利用者個人のカード状態そのものは共有しません。
* `state=1` のように指定した場合、その閲覧者のLocalStorage上で状態1のカードだけが表示されます。
* 他人のLocalStorage状態は共有されません。
* SNS投稿やスクショ用表示では、共有URLよりも画像・投稿文生成の方が分かりやすい場合があります。
* URLクエリ、LocalStorage、初期表示条件の優先順位は、実装前に整理します。

## アクセシビリティ上の注意

* フィルタボタンの選択状態を `aria-pressed` などで伝えます。
* フィルタパネルを開閉する場合は `aria-expanded`、`aria-controls` を使います。
* 絞り込み結果件数をテキストで表示します。
* 結果件数の変化を必要に応じて `aria-live` で伝えます。
* 色だけでフィルタ状態を示しません。
* 赤枠・水色枠フィルタにもテキストラベルを併記します。
* キーボード操作でフィルタを変更できるようにします。
* フィルタ結果が0件の場合も、空白だけにしません。
* タッチ操作でも押しやすいサイズと間隔を確保します。
* フォーカスインジケータを消さず、現在位置が分かるようにします。

## 実装時の注意

今回のPRでは実装しませんが、今後の実装時は次の点に注意します。

* 既存のカードクリック挙動を壊さないようにします。
* 既存LocalStorageキー `sanga-schedule-button-states-v1` を変更しません。
* フィルタ条件は別キーで保存します。
* ID `sec01`〜`sec38` を維持します。
* フィルタで非表示にするだけで、DOMから完全削除するか、`hidden` 属性で隠すかを検討します。
* スクリーンリーダー向けには非表示カードの扱いに注意します。
* JSONの未知キーがあっても壊れないようにします。
* 初期表示速度を落とさないようにします。
* フィルタ条件が増えてもスマホ画面で操作しやすくします。
* CSSだけでなく、JS側で表示件数・空状態を管理します。
* 開催年判定は、既存の開催日・候補日・節番号から判定するロジックと整合させます。
* `is_visible` が将来導入された場合、非表示扱いの試合をフィルタで誤って表示しないようにします。

## やってはいけないこと

* フィルタリング機能のために、スプレッドシートや公開JSONの正本データを閲覧者ごとに変更しないこと。
* 利用者ごとの状態、ラベル、メモ、チケット購入済み状態をスプレッドシートや公開JSONに保存しないこと。
* 赤枠・水色枠の意味をページ側で固定しないこと。
* `赤枠 = 参戦予定` のような固定表現をUIやドキュメントの基本方針にしないこと。
* 既存LocalStorageキー `sanga-schedule-button-states-v1` を理由なく変更しないこと。
* 既存の `0 / 1 / 2` の保存構造を否定しないこと。
* フィルタで非表示になったカードのLocalStorage状態を削除しないこと。
* ID `sec01`〜`sec38` を不用意に変更しないこと。
* 画面上で色だけに依存して選択状態やフィルタ状態を伝えないこと。
* 認証情報、サービスアカウントJSON、APIキー、`.env`、Secretsの実値を作成・保存・記載しないこと。
