# フィルタリング機能の設計

## この設計の目的

年間スケジュールページに、目的の試合だけを探しやすくするフィルタリング機能を追加するための設計を整理します。

年間スケジュールページは、スマートフォンで京都サンガF.C.の試合予定を確認し、閲覧者ごとのカード状態をLocalStorageに保存し、SNS投稿やスクショ用表示へつなげることを主な目的とします。今後、試合数が増えたり、カップ戦・ACL戦が追加されたり、利用者ごとの色枠状態やラベルが増えたりすると、必要な試合だけを絞り込めることが重要になります。

この文書では、実装前に次の点を整理します。

* フィルタ対象にする情報
* 初期実装で扱うフィルタ
* 将来拡張で扱うフィルタ
* パーソナライズ状態との関係
* UI方針
* LocalStorage保存方針
* 共有URLとの関係
* アクセシビリティ上の注意

今回の設計はドキュメント整理であり、HTML、CSS、JavaScript、JSONの変更を前提にしません。

## フィルタリングの基本方針

* スマートフォンで目的の試合を素早く探せることを重視します。
* 初期表示では全試合を表示します。
* フィルタを選ぶと、該当する試合だけを表示します。
* フィルタ解除で全試合表示に戻せるようにします。
* 表示されないカードのLocalStorage状態は削除しません。
* フィルタは閲覧者ごとの表示補助であり、スプレッドシートやJSONの正本データを変更しません。
* フィルタ条件は、必要に応じてLocalStorageに保存します。
* 将来的に共有URLでフィルタ条件を指定できる余地を残します。
* フィルタ結果は、SNS投稿、スクショ用表示、カレンダー連携などの対象選択にも使えるようにします。

## フィルタ対象

フィルタ対象は、全閲覧者に共通する「データ由来フィルタ」と、閲覧者ごとに変わる「LocalStorage由来フィルタ」に分けます。

### データ由来フィルタ

`public/data/matches.json` や、将来的なGoogleスプレッドシート由来の項目です。試合そのものの属性であり、閲覧者ごとに変わらない情報を対象にします。

候補は次のとおりです。

| フィルタ対象 | 対応するJSONキー候補 | 用途 |
| --- | --- | --- |
| 大会 | `competition` / `competition_label` | J1、ルヴァンカップ、天皇杯、ACL、親善試合などで絞り込む。 |
| 開催年 | `match_date` / `date_candidates` | 2026年、2027年など、年またぎシーズン内の試合を絞り込む。 |
| HOME / AWAY | `home_away` / `home_away_label` | ホーム、アウェイ、中立地、未定を絞り込む。 |
| 試合状態 | `match_status` | 試合前、試合中、試合終了、延期、中止などで絞り込む。 |
| 結果 | `result` | 勝、分、敗などで絞り込む。 |
| 表示フラグ | `is_visible` | 公開画面に表示する試合かどうかを制御する。 |
| 対戦相手 | `opponent` / `opponent_code` | 特定の対戦相手を絞り込む。 |
| 会場 | `venue` | サンガスタジアム、アウェイ会場などで絞り込む。 |
| 日程確定状態 | `status` / `date_candidates` | 確定、未確定、候補日ありなどで絞り込む。 |

### LocalStorage由来フィルタ

閲覧者ごとの状態です。スプレッドシートや公開JSONには保存せず、端末内のLocalStorageに保存された個人状態を対象にします。

候補は次のとおりです。

| フィルタ対象 | 現時点の扱い | 備考 |
| --- | --- | --- |
| 未選択 | 実装済みカード状態 `0` | 枠線なしの通常状態。 |
| 赤色枠 | 実装済みカード状態 `1` | 意味は固定せず、利用者が自由に決める色枠として扱う。 |
| 水色枠 | 実装済みカード状態 `2` | 意味は固定せず、利用者が自由に決める色枠として扱う。 |
| 将来追加される色枠 | 将来機能 | 状態値や表示色を増やす場合に検討する。 |
| 利用者が設定したラベル | 将来機能 | 「参戦予定」などの意味をページ側で固定せず、利用者設定として扱う。 |
| メモあり | 将来機能 | 個人メモ機能が追加された場合に検討する。 |
| チケット購入済み | 将来機能 | 閲覧者個人の準備状態としてLocalStorageで扱う。 |

現時点で実装済みなのは、カード状態 `0 / 1 / 2` のみです。利用者ラベル、メモ、チケット購入済みなどは将来機能として扱います。

## 初期実装で扱うフィルタ

初期実装では、実装しやすく、スマートフォンで効果が大きいものに絞ります。

推奨する初期フィルタは次のとおりです。

```text
すべて
HOME
AWAY
2026
2027
未確定日程
枠線あり
赤色枠
水色枠
```

各フィルタの方針は次のとおりです。

| フィルタ | 判定方針 |
| --- | --- |
| すべて | フィルタ解除として扱い、全試合を表示する。 |
| HOME | `home_away` が `H` の試合を対象にする。 |
| AWAY | `home_away` が `A` の試合を対象にする。 |
| 2026 | 開催日、候補日、節番号から判定する既存ロジックと整合させる。 |
| 2027 | 開催日、候補日、節番号から判定する既存ロジックと整合させる。 |
| 未確定日程 | `status: tentative` または候補日ありの試合を対象にする。 |
| 枠線あり | LocalStorage状態が `1` または `2` のカードを対象にする。 |
| 赤色枠 | LocalStorage状態 `1` のカードを対象にする。 |
| 水色枠 | LocalStorage状態 `2` のカードを対象にする。 |

赤色枠・水色枠の意味は固定しません。利用者が自分で意味を決める色枠として扱います。

## 将来拡張で扱うフィルタ

将来拡張では、試合数の増加、戦績更新、個人設定の拡張、SNS投稿やスクショ用表示との連携を見据えて、次のフィルタを検討します。

| フィルタ | 方針 |
| --- | --- |
| 大会別 | カップ戦・ACL戦を追加した場合は、大会別フィルタが重要になる。 |
| 試合状態別 | 試合前、試合中、試合終了、延期、中止などで絞り込む。 |
| 結果別 | 戦績更新後は、勝・分・敗の結果フィルタが使える。 |
| 対戦相手別 | 特定クラブとの試合を探しやすくする。 |
| 会場別 | ホームスタジアム、遠征先、未定会場などで絞り込む。 |
| 利用者ラベル別 | 利用者ラベル設定が実装されたら、赤色枠・水色枠ではなくラベル名でのフィルタも検討する。 |
| メモあり | 追加メモ機能が実装されたら、メモありフィルタを検討する。 |
| チケット購入済み | チケット購入状態をLocalStorageで持つ場合に検討する。 |
| カレンダー登録対象 | カレンダー連携で登録したい試合の選択に使う。 |
| スクショ対象 | SNS投稿やスクショ用表示で、フィルタ結果をそのまま共有対象にできるようにする。 |

SNS投稿やスクショ用表示では、フィルタ結果をそのまま共有対象にできるようにします。ただし、共有されるべき内容と、閲覧者の端末内だけに保存される個人状態は分けて扱います。

## パーソナライズ状態との関係

`docs/personalization.md` の方針に合わせ、色枠の意味はページ側で固定しません。

* フィルタでは、まずは `赤色枠`、`水色枠` のように見た目上の状態として扱います。
* 赤色枠・水色枠に用途を固定する表現は避けます。
* 将来的に利用者ラベルが設定された場合は、ラベル名でフィルタ表示できるようにします。
* LocalStorage状態は、フィルタで非表示になっても保持します。
* 保存内容削除ボタンでは、フィルタ条件も削除対象にするか検討します。
* フィルタ条件とカード状態は別のLocalStorageキーに分ける方針がよいです。
* 利用者ごとの状態やラベルは、スプレッドシートや公開JSONに保存しません。

既存のカード状態保存キーは変更しません。

```text
sanga-schedule-button-states-v1
```

将来的なフィルタ条件保存キー案は次のとおりです。

```text
sanga-schedule-filter-settings-v1
```

保存イメージは次のとおりです。

```json
{
  "activeFilters": {
    "homeAway": "H",
    "year": "2026",
    "personalState": "1"
  }
}
```

## UI方針

表示関連の操作は、日程表示欄をできるだけシンプルに保つことを優先して設計します。スマートフォンでは画面幅が限られるため、日程カード上部やカード付近にフィルタ、表示列、色枠設定などの操作を常時大きく並べる構成は避けます。

基本方針は次のとおりです。

* 日程表示欄の上部やカード付近には、フィルタUIを常時大きく表示しません。
* 日程表示欄は、できるだけ日程カードの表示に専念させます。
* 表示列切替、フィルタリング、表示モード切替、色枠設定などの表示関連操作は、フッター付近の操作エリアへ集約します。
* フッター付近に「歯車アイコン + 設定」ボタンを配置し、そこから設定パネルを開く案を基本方針にします。
* 設定ボタンはアイコンだけにせず、必ず「設定」テキストを併記します。
* 設定パネル内で、表示列、フィルタ、表示モード、パーソナライズ状態の色・ラベル設定などを扱えるようにします。
* 初期実装では、設定パネル内の機能を段階的に追加します。
* 日程カード上部にフィルタチップを多数常時表示する案は、画面がごちゃつくため優先度を下げます。
* ただし、将来的に選択中フィルタの要約だけを小さく表示する余地は残します。
* 選択中フィルタ、表示件数、フィルタ結果0件時のメッセージは、日程表示を妨げない最小限の状態表示として検討します。

### 設定パネルへ集約する候補

設定パネルは、最初からすべての機能を入れるのではなく、既存機能と初期フィルタリングから段階的に育てます。

| 候補機能 | 初期段階の扱い | 将来方針 |
| --- | --- | --- |
| 表示列切替 | 既存の2列 / 3列 / 4列切替を中心に扱う。短期的には現状UI維持も許容する。 | 設定パネル内の「表示」区分へ移す。 |
| フィルタリング | 初期フィルタの追加時に、設定パネル内へ入れることを基本にする。 | 大会、状態、結果、会場、個人ラベルなどへ拡張する。 |
| 表示モード切替 | 初期実装では必須にしない。 | 月別、状態別、スクショ用などの切替と連携する。 |
| パーソナライズ状態の色・ラベル設定 | 今回は実装しない。 | `docs/personalization.md` の方針に沿って、利用者が色枠数やラベルを設定できる候補として設定パネルへ入れる。 |
| スクショ用表示 | 今回は実装しない。 | フィルタ結果や個人状態と連携して、SNS共有しやすい表示へ切り替える。 |
| SNS投稿補助 | 今回は実装しない。 | フィルタ結果や選択状態から投稿文・画像生成を補助する。 |
| LocalStorage保存内容削除 | 既存の使い方パネル内の削除ボタンは急に廃止しない。 | 設定パネル内の「保存状態」区分へ移すか、使い方パネル内に残すかを検討する。 |

初期段階では、既存の表示列切替と今後追加するフィルタリングを中心に扱います。色枠設定、利用者ラベル設定、スクショ用表示、SNS投稿補助は、設計上の候補として整理するだけで、この段階では実装しません。

### 現在のフッター操作との関係

現在のページには、フッター付近に「使い方」ボタン、表示列切替、保存内容削除ボタンがあります。これらはすぐに廃止せず、段階的に設定パネルへ集約する方針にします。

#### 短期方針

* 既存UIを急に大きく変えません。
* 使い方ボタンは維持します。
* 表示列切替は、当面は現状のままでもよいものとします。
* 保存内容削除ボタンは、当面は使い方パネル内に残すことを許容します。
* フィルタリング初期実装時に、設定パネルへ集約するか、既存のフッター操作から段階的に移行するかを判断します。

#### 中期方針

* フッター付近に「歯車アイコン + 設定」ボタンを追加します。
* 設定パネル内に「表示」「絞り込み」「保存状態」などの区分を置きます。
* 表示列切替を設定パネル内へ移します。
* フィルタリングを設定パネル内へ入れます。
* 保存内容削除も設定パネル内へ移すか、使い方パネル内に残すかを検討します。

#### 長期方針

* 色枠設定、利用者ラベル設定、スクショ用表示、SNS投稿補助も設定パネルと連携します。
* 日程表示欄は、日程カードと必要最小限の状態表示だけにします。
* 表示関連機能が増えても、カード一覧の読みやすさを優先します。

### 設定パネルのUI案

設定パネルは、既存の使い方パネルに近い開閉UIとして実装してもよいです。スマートフォンでは、画面下から開くボトムシート風UIも検討します。

UI案は次のとおりです。

* ボタンは「⚙ 設定」のように、歯車アイコンと「設定」テキストを併記します。
* アイコンだけのボタンにはしません。
* ボタンには、何を開くのか分かるラベルを付けます。
* 設定パネル内は、見出しごとに分けます。
* すべての操作を一度に表示しすぎず、初期実装では必要な区分から追加します。
* パネル内で変更した結果が分かるように、選択状態と表示件数をテキストでも示します。

構成例は次のとおりです。

```text
設定
├─ 表示
│  ├─ 2列
│  ├─ 3列
│  └─ 4列
├─ 絞り込み
│  ├─ すべて
│  ├─ HOME
│  ├─ AWAY
│  ├─ 2026
│  ├─ 2027
│  ├─ 未確定
│  ├─ 枠線あり
│  ├─ 赤色枠
│  └─ 水色枠
├─ 保存状態
│  └─ このページの保存内容を削除
└─ 今後追加
   ├─ 色枠設定
   ├─ SNS投稿
   └─ スクショ用表示
```

### 設定パネル化する場合のアクセシビリティ方針

設定パネルへ表示関連操作を集約する場合も、キーボード操作、スクリーンリーダー、タッチ操作で分かりやすく使えることを前提にします。

* 設定ボタンには、歯車アイコンだけでなく「設定」テキストを併記します。
* 設定ボタンには `aria-expanded` と `aria-controls` を使います。
* 設定パネルは、内容と表示方法に応じて `role="dialog"` または適切なランドマークを検討します。
* パネルを開いたら、フォーカスをパネル内の見出しまたは最初の操作へ移します。
* パネルを閉じたら、設定ボタンへフォーカスを戻します。
* Escキーや閉じるボタンで閉じられるようにします。
* 表示列やフィルタの選択状態は `aria-pressed` などで伝えます。
* 絞り込み件数はテキストで表示します。
* 結果件数の変化は、必要に応じて `aria-live` で伝えます。
* 色だけで選択状態やフィルタ状態を伝えません。
* 赤色枠・水色枠フィルタにもテキストラベルを併記し、色の意味は固定しません。
* タッチ操作でも押しやすいサイズと間隔を確保します。
* フォーカスインジケータを消さず、現在位置が分かるようにします。

## LocalStorage保存方針

フィルタ条件は、必要に応じてLocalStorageへ保存します。ただし、カード状態とは別キーで保存します。

* 既存のカード状態保存キー `sanga-schedule-button-states-v1` は変更しません。
* フィルタ条件用には `sanga-schedule-filter-settings-v1` のような別キーを検討します。
* フィルタ条件を保存する場合でも、スプレッドシートや公開JSONは変更しません。
* フィルタで非表示になったカードの状態は削除しません。
* フィルタ条件が壊れている、または未知の値を含む場合は、安全側に倒して全試合表示へ戻せるようにします。
* 保存内容削除ボタンでカード状態だけを消すのか、フィルタ条件も消すのかはUI文言とあわせて検討します。
* 個人ラベル、メモ、チケット購入済みなどが追加された場合も、公開JSONではなくLocalStorage側に保存します。

## 共有URLとの関係

将来的に共有URL機能を追加する場合に備えて、フィルタ条件をURLクエリで表現できる余地を残します。

例:

```text
?homeAway=H
?year=2026
?status=tentative
?state=1
?competition=J1
```

方針は次のとおりです。

* URLで共有するのはフィルタ条件です。
* 利用者個人のカード状態そのものは共有しません。
* `state=1` のように指定した場合、その閲覧者のLocalStorage上で状態1のカードだけが表示されます。
* 他人のLocalStorage状態は共有されません。
* SNS投稿やスクショ用表示では、共有URLよりも画像・投稿文生成の方が分かりやすい場合があります。
* URLクエリ、LocalStorage、初期表示条件の優先順位は、実装前に整理します。

## アクセシビリティ上の注意

* フィルタボタンの選択状態を `aria-pressed` などで伝えます。
* フィルタパネルを開閉する場合は `aria-expanded`、`aria-controls` を使います。
* 絞り込み結果件数をテキストで表示します。
* 結果件数の変化を必要に応じて `aria-live` で伝えます。
* 色だけでフィルタ状態を示しません。
* 赤色枠・水色枠フィルタにもテキストラベルを併記します。
* キーボード操作でフィルタを変更できるようにします。
* フィルタ結果が0件の場合も、空白だけにしません。
* タッチ操作でも押しやすいサイズと間隔を確保します。
* フォーカスインジケータを消さず、現在位置が分かるようにします。

## 実装時の注意

今回のPRでは実装しませんが、今後の実装時は次の点に注意します。

* 既存のカードクリック挙動を壊さないようにします。
* 既存LocalStorageキー `sanga-schedule-button-states-v1` を変更しません。
* フィルタ条件は別キーで保存します。
* ID `sec01`〜`sec38` を維持します。
* フィルタで非表示にするだけで、DOMから完全削除するか、`hidden` 属性で隠すかを検討します。
* スクリーンリーダー向けには非表示カードの扱いに注意します。
* JSONの未知キーがあっても壊れないようにします。
* 初期表示速度を落とさないようにします。
* フィルタ条件が増えてもスマホ画面で操作しやすくします。
* CSSだけでなく、JS側で表示件数・空状態を管理します。
* 開催年判定は、既存の開催日・候補日・節番号から判定するロジックと整合させます。
* `is_visible` が将来導入された場合、非表示扱いの試合をフィルタで誤って表示しないようにします。

## やってはいけないこと

* フィルタリング機能のために、スプレッドシートや公開JSONの正本データを閲覧者ごとに変更しないこと。
* 利用者ごとの状態、ラベル、メモ、チケット購入済み状態をスプレッドシートや公開JSONに保存しないこと。
* 赤色枠・水色枠の意味をページ側で固定しないこと。

* フィルタで非表示になったカードのLocalStorage状態を削除しないこと。

* 認証情報、サービスアカウントJSON、APIキー、`.env`、Secretsの実値を作成・保存・記載しないこと。

## 大会フィルタ

2026-06-24時点で、設定パネルの「絞り込み」に大会フィルタを追加しました。

- 対象は `J1`、`天皇杯`、`ルヴァン杯` です。
- 既存フィルタと同じく、日程表示欄ではなく設定パネルから操作します。
- 内部値は `competition-j1`、`competition-emp`、`competition-lev` を使います。
- 判定対象は `matches.json` の `competition` 由来で、`J1`、`EMP`、`LEV` に対応します。
- フィルタ状態は既存のLocalStorageフィルタ設定 `sanga-schedule-filter-settings-v1` に保存します。新しいLocalStorageキーは追加しません。
- 共有画像生成にも、現在の大会フィルタを含む絞り込み結果が反映されます。
- 大会フィルタ適用時も、対象試合がない年の見出しだけが残らないように、既存の年見出し表示制御を利用します。

今後フィルタ処理を整理する場合は、HTMLの `data-filter`、`public/assets/app.js` の有効値・ラベル・判定処理、`tools/validate-app-contract.js` の契約チェック、このドキュメントの説明を同時に確認します。

## 2026-06-24 フィルタ処理の内部構造

現在のフィルタは単一選択方式です。`activeFilter` に現在のフィルタ値を保持し、`normalizeFilter()` で `VALID_FILTERS` に含まれる値だけを有効にします。不明な保存値や壊れたLocalStorage値は既定値 `all` に戻します。

`public/assets/app.js` では、フィルタ値、表示ラベル、判定種別をフィルタ定義として管理します。互換性維持のため、`VALID_FILTERS` と `FILTER_LABELS` は引き続き契約として残します。

大会フィルタは次の対応で扱います。

| フィルタ値 | 表示ラベル | JSON `competition` |
| --- | --- | --- |
| `competition-j1` | `J1` | `J1` |
| `competition-emp` | `天皇杯` | `EMP` |
| `competition-lev` | `ルヴァン杯` | `LEV` |

フィルタ適用時は、試合カードの `hidden`、対象試合がない年見出し、件数表示、0件メッセージ、フィルタボタンの `aria-pressed` を更新します。共有画像生成は現在表示されているDOMを画像化するため、フィルタ後のカード表示状態と年見出し表示状態が引き続き反映されます。
