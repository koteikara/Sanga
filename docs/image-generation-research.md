# 画像生成ライブラリ導入可否調査

## この調査の目的

年間スケジュールページのスクショ用表示を、ブラウザ上でPNGなどの画像として保存・共有できるようにするため、画像生成ライブラリの導入可否を整理します。

今回の調査はドキュメント整理のみを目的とし、実装、依存関係追加、CDN script追加、公開HTML/CSS/JavaScript変更、公開JSON変更は行いません。

## 現在の課題

現在の年間スケジュールページにはスクショ用表示がありますが、共有時には次の課題があります。

* スクショ用表示はあるものの、日程数が多いためページ全体が縦に長くなります。
* スマートフォン標準のスクリーンショットでは、画面内に表示されている範囲しか1枚に収まりません。
* iOS、Android、ブラウザごとにフルページスクリーンショットの方法が異なります。
* 利用者へフルページスクリーンショットの操作手順を説明しても、操作が分かりにくく使われにくい可能性があります。
* SNS投稿では、日程表示を1枚画像として共有できる方が見た人に伝わりやすくなります。
* ブラウザ内で画像化できれば、フルページスクリーンショット操作を知らない利用者でも保存・共有しやすくなります。

## 実現したいこと

初期実装で目指すことは次のとおりです。

* 現在の絞り込み結果を画像化する。
* スクショ用表示の見た目を画像化する。
* 表示列 `1列 / 2列 / 3列 / 4列` を反映する。
* 赤色枠 / 水色枠の状態を反映する。
* 年見出しを反映する。
* 短い注意書きと作成者表記を含める。
* 画像をダウンロードできる。
* 可能ならスマートフォンで共有しやすくする。

初期実装では、次のことは行わない方針にします。

* X API連携。
* 自動投稿。
* ログイン機能。
* サーバー側画像生成。
* 外部サーバーへの画像送信。
* 個人情報の収集。

## 調査対象ライブラリ

主な調査対象は次の5つです。

* `html2canvas`
  * 方式: DOMを解析してCanvasへ描画する方式です。実際のスクリーンショットではなく、DOM情報から画像を再構成します。
  * ライセンス: MIT Licenseです。
  * 導入: npm導入、または配布ファイルをCDN等から読み込む運用が候補になります。
  * 出力: Canvas経由でPNG/JPEGなどへ変換できます。
  * 特徴: 利用例が多く情報を探しやすい一方、CSS再現性は実ブラウザ描画そのものではないため検証が必要です。
  * 注意: 外部画像はCORS設定次第でCanvasが汚染され、出力できなくなる可能性があります。
  * 参照: <https://github.com/niklasvh/html2canvas>

* `html-to-image`
  * 方式: 対象DOMをSVGの `foreignObject` などを経由してCanvasへ描画し、PNG/SVG/JPEG等を生成する方式です。
  * ライセンス: MIT Licenseです。
  * 導入: npm導入が基本ですが、CDN経由での配布利用も候補になります。
  * 出力: PNG、JPEG、SVG、Blob、Canvasなどが候補です。
  * 特徴: `dom-to-image` 系の流れを汲み、DOMノードから画像を作る用途に特化しています。
  * 注意: SVG `foreignObject` やWebフォント、外部画像、Safari系の挙動を実機で確認する必要があります。
  * 参照: <https://github.com/bubkoo/html-to-image>

* `dom-to-image-more`
  * 方式: `dom-to-image` のフォークで、SVGまたはCanvas経由でPNG/JPEG/SVGを生成します。
  * ライセンス: MIT Licenseです。
  * 導入: npm導入、または配布ファイルの静的配置が候補になります。
  * 出力: PNG、JPEG、SVGなどが候補です。
  * 特徴: 元の `dom-to-image` に対して修正が取り込まれており、現時点では `dom-to-image` より候補にしやすいです。
  * 注意: 長尺DOM、Webフォント、外部画像、Safari系の実用性は検証が必要です。
  * 参照: <https://github.com/1904labs/dom-to-image-more>

* `dom-to-image`
  * 方式: DOMノードをSVGまたはCanvas経由でPNG/JPEG/SVG化する方式です。
  * ライセンス: MIT Licenseです。
  * 導入: npm導入やCDN利用は可能ですが、現在は後継・フォーク候補との比較が必要です。
  * 出力: PNG、JPEG、SVGなどが候補です。
  * 特徴: この系統の元になった有名ライブラリです。
  * 注意: メンテナンス停滞が懸念されるため、新規導入候補としては優先度を下げます。
  * 参照: <https://github.com/tsayen/dom-to-image>

* `modern-screenshot`
  * 方式: `html-to-image` のフォークで、DOMノードからHTML5 CanvasとSVGを使って画像を生成します。
  * ライセンス: MIT Licenseです。
  * 導入: npm導入が基本候補です。
  * 出力: PNG、JPEG、SVG、Blob、Canvasなどが候補です。
  * 特徴: 比較的新しい候補で、TypeScriptでの利用も想定されています。
  * 注意: 静的サイトへ入れる場合のバンドル方法、CDN利用可否、スマホSafariでの長尺出力を確認する必要があります。
  * 参照: <https://github.com/qq15725/modern-screenshot>

必要に応じて追加候補として、サーバー側でPuppeteer等を使う `node-html-to-image` 系も考えられます。ただし、初期実装ではサーバー側画像生成と外部サーバー送信を行わない方針のため、今回の優先候補からは外します。

## 比較観点

導入可否は次の観点で比較します。

* 現在もメンテナンスされているか。
* ライセンスが静的サイト運用で扱いやすいか。
* npm導入が必要か。
* CDN利用が可能か。
* ブラウザだけで動くか。
* TypeScript前提ではなく素のJavaScriptから使えるか。
* 依存関係が重すぎないか。
* PNG出力ができるか。
* SVG経由かCanvas経由か。
* CSS Grid、疑似要素、グラデーション、box-shadow、borderなどのCSS再現性。
* Webフォント対応。
* 外部画像やCORSの扱い。
* 長いDOMを画像化できるか。
* スマートフォンブラウザで実用的な速度・メモリ使用量か。
* Safari / iOS Safari で動作するか。
* GitHub Pages / スターレンタルサーバーの静的サイトで使えるか。

## このページでの技術的な懸念

年間スケジュールページ固有の要素と懸念は次のとおりです。

* CSS Grid
  * 表示列 `1列 / 2列 / 3列 / 4列` の再現に直結します。
  * ライブラリによってはGridの計算結果を正しく画像化できるか確認が必要です。
* 疑似要素
  * `::before` / `::after` に依存した装飾がある場合、画像化に含まれるか確認が必要です。
* グラデーション背景
  * 背景の見た目がSNS画像の印象に影響するため、再現性を確認します。
* `box-shadow`
  * カードの立体感や区切りが失われないか確認します。
* `border`
  * 赤色枠 / 水色枠の状態に関わるため、線幅、角丸、色が反映されるか確認します。
* stickyボタン
  * スクショ用表示の「通常表示に戻る」ボタンは、共有画像には含めない方が自然です。
  * 画像化対象DOMから外す、または画像化前に一時的に非表示にする必要があります。
* `hidden`属性
  * 非表示カード、非表示パネル、折りたたみUIが画像に入らないか確認します。
* LocalStorage状態
  * 表示列、フィルタ、赤色枠 / 水色枠状態を反映する一方、LocalStorageの中身そのものは画像や投稿文に出しません。
* フィルタ表示
  * 現在表示されているカードだけを画像化し、非表示カードが混ざらないことを確認します。
* 年見出し
  * フィルタ後に該当カードがある年だけ自然に表示されるか確認します。
* 長い縦スクロール
  * 全38試合や複数年分を画像化すると、CanvasやSVGの高さ制限、メモリ制限、処理時間の問題が起きる可能性があります。
* スマホ幅前提
  * 4列表示では1カードあたりの幅が小さくなり、画像化後に文字が読みにくくなる可能性があります。

特に確認すべき点は次のとおりです。

* CSS Gridが画像に正しく反映されるか。
* `box-shadow` や `border` が再現されるか。
* グラデーション背景が再現されるか。
* 疑似要素が再現されるか。
* `hidden` 要素が画像に入らないか。
* stickyの「通常表示に戻る」ボタンを画像に含めない方法を用意できるか。
* 画像化対象範囲を `.phone` 全体にするか、`.json-preview` 周辺に限定するか。
* 長い画像の高さ制限やメモリ問題が発生しないか。
* iPhone Safariで巨大Canvasが失敗しないか。
* 4列表示で小さすぎる場合に、画像用の最大列数や文字サイズ調整が必要か。
* 画像化前に一時的に設定ボタン、戻るボタン、操作UIを隠す必要があるか。

## 画像化対象範囲の候補

### 案A: `.phone` 全体を画像化

メリット:

* 見た目全体をそのまま画像化しやすいです。
* 作成者表記や注意書きも含めやすいです。
* 既存のスクショ用表示に近い範囲をそのまま扱えます。

デメリット:

* 通常表示に戻るボタンや不要UIが入りやすくなります。
* 余白や背景が大きくなり、SNS投稿時に主役の日程が小さく見える可能性があります。
* スマホ画面全体の再現に近くなり、画像の縦横サイズが大きくなりやすいです。

### 案B: `.json-preview` + 作成者表記 + 注意書きを別ラッパーにして画像化

メリット:

* 必要な範囲だけを画像化しやすいです。
* SNS投稿向けに余白、注意書き、作成者表記を整えやすいです。
* 戻るボタンや設定UIなど、共有画像に不要なUIを避けやすくなります。
* 既存の表示状態を活かしながら、画像化対象を限定できます。

デメリット:

* 専用ラッパー設計が必要です。
* 既存DOMの組み替え、または画像化時だけのラッパー追加が必要になる可能性があります。
* ラッパー内外でCSSの継承や余白が変わらないか確認が必要です。

### 案C: 画像生成専用の非表示DOMを作る

メリット:

* SNS画像として最適化しやすいです。
* 通常表示に影響しにくく、余白、列数、見出し、注意書きを画像専用に調整できます。
* 画像の縦横比やカード密度を制御しやすくなります。

デメリット:

* DOMを二重管理しやすくなります。
* フィルタ、赤色枠 / 水色枠、年見出し、表示列との同期が複雑になります。
* 通常表示と画像表示で内容差分が出るリスクがあります。

現時点の推奨は、まず案Bを優先して検討することです。案Bは、既存の表示結果を活かしつつ、共有画像に不要なUIを除外しやすく、案Cほど二重管理になりにくいためです。案Aは最小検証には向きますが、SNS向け画像としては不要UIや余白の問題が出やすいと考えます。案Cは将来的な最適化候補として残し、初期実装では避けるのが安全です。

## アクセシビリティ上の注意

画像生成機能は補助機能として扱い、HTMLの日程表示を主とする方針を維持します。

* 画像生成は補助機能であり、HTMLの日程表示を主とします。
* 画像だけでなく、投稿文生成・コピーも併用できるようにします。
* X投稿時は、本文や代替テキストを併記することを促します。
* 画像内の文字サイズが小さくなりすぎないようにします。
* 4列画像では可読性が落ちる可能性があるため、注意書きや列数調整を検討します。
* 生成完了や失敗は `aria-live` で通知します。
* 生成ボタン・保存ボタンはキーボード操作可能な `button` として実装します。
* 処理中状態をスピナーだけでなくテキストでも示します。
* 色だけで赤色枠 / 水色枠の状態を伝えない配慮を維持します。
* 生成に時間がかかる場合でも、フォーカスを失わせたり、操作不能な状態を説明なしに作ったりしないようにします。

## セキュリティ・プライバシー上の注意

初期実装では、画像生成をブラウザ内で完結させる方針にします。

* 画像生成はブラウザ内で完結させます。
* 外部サーバーへDOMや画像を送信しません。
* X APIなど外部API連携は初期実装では行いません。
* 認証情報やAPIキーを使いません。
* 外部CDNを使う場合は、供給元リスク、改ざんリスク、可用性、SRI設定を確認します。
* npmで同梱する場合は、依存関係、ライセンス、更新状況を確認します。
* 生成画像に個人情報を含めない設計にします。
* LocalStorageの中身を画像に直接出しません。
* 赤色枠 / 水色枠の意味は固定しません。
* 画像化対象DOMに、非表示の設定値、デバッグ情報、不要な属性値が含まれないように確認します。

## 導入しない場合の代替策

画像生成ライブラリを導入しない場合の代替策は次のとおりです。

* スクショ用表示の案内を改善し、iOS / Android / Chrome / Safari別のフルページスクリーンショット手順を短く案内する。
* スクショ用表示をさらに短くし、フィルタや列数を工夫してスマホ画面内に収まりやすくする。
* 投稿文生成・コピー機能を先に追加し、画像なしでも共有しやすくする。
* 1年分、HOMEのみ、AWAYのみなど、短い範囲に分けて共有できる導線を強化する。
* 印刷/PDF向け表示を整備し、OS標準のPDF保存や共有に任せる。

ただし、フルページスクリーンショット操作だけに頼ると利用されにくい懸念は残るため、画像生成ライブラリの小規模検証は行う価値があります。

## 導入形態の候補

### CDNで読み込む

メリット:

* 導入が簡単です。
* ビルド環境が不要です。
* ローカル検証の初期段階では試しやすいです。

デメリット:

* 外部CDNに依存します。
* セキュリティ、可用性、SRI確認が必要です。
* 公共向け・長期運用の感覚では避けたい可能性があります。
* 本番閲覧時に外部通信が増える可能性があります。

### npmで導入して同梱

メリット:

* バージョンを固定しやすいです。
* 外部通信なしで動かせます。
* GitHub Pages / 本番静的配信と相性がよいです。
* ライセンスや依存関係を管理しやすくなります。

デメリット:

* `package.json` 管理が必要です。
* ビルドやバンドル手順が必要になる可能性があります。
* 現在のシンプルな静的構成が複雑になります。
* 依存関係更新や脆弱性確認の運用が必要になります。

### ライブラリファイルをvendor配下に配置

メリット:

* ビルド不要で静的配信できます。
* 外部CDNに依存しません。
* 現在の静的サイト運用に比較的近い形で扱えます。

デメリット:

* 更新管理が手動になります。
* ライセンス表記が必要です。
* minifiedファイルの扱い、出所、改ざん確認に注意が必要です。
* 依存関係があるライブラリでは単一ファイル配置が難しい場合があります。

現時点の推奨は、まずnpm導入より前に、ローカル検証で候補を絞る方針です。静的ページ運用を維持したいため、いきなり依存関係を追加せず、候補ライブラリごとに最小HTMLまたはローカル検証ブランチでスクショ用表示相当のDOMを画像化し、実用性を確認してから導入形態を決めます。

## 小規模検証案

実装PRへ進む前に、次の内容を小さく検証します。

検証対象:

* スクショ用表示のDOMをPNG化できるか。
* `1列 / 2列 / 3列 / 4列` で崩れないか。
* 赤色枠 / 水色枠が反映されるか。
* 年見出しが反映されるか。
* 長い画像を生成できるか。
* iPhone Safariで動くか。
* Android Chromeで動くか。
* PC Chromeで動くか。
* 生成時間が許容範囲か。
* 画像サイズが大きすぎないか。
* ダウンロードできるか。
* Web Share APIと組み合わせられるか。

検証方法:

* 一時的なローカル検証ブランチで試します。
* 依存追加は検証PRではなくローカルのみで行います。
* `public/data/matches.json` は変更しません。
* 実装PRに進む前に候補ライブラリを1つに絞ります。
* まず案B相当の限定ラッパーで検証し、必要なら案Aで差分を確認します。
* 生成対象件数を少数、1年分、全件の3段階に分け、処理時間と失敗率を確認します。
* iPhone Safariでは、長尺Canvasが失敗しないか、真っ白画像や途中で切れる画像にならないかを重点確認します。

## 推奨方針

小規模検証は `experiments/image-generation/` で行い、優先検証候補は `html-to-image` と `modern-screenshot` とします。

フルページスクリーンショット操作だけに頼ると利用されにくいため、画像生成機能の導入検討は妥当です。

ただし、年間スケジュールページは長尺DOM、スマートフォンSafari、CSS Grid、疑似要素、グラデーション、赤色枠 / 水色枠、LocalStorage由来の表示状態など、画像化で差分が出やすい要素を含みます。依存関係、ライセンス、外部CDN、セキュリティ、プライバシーの懸念もあるため、即実装は避けます。

まずは候補ライブラリを比較し、ローカルで小規模検証します。初期実装では、外部サーバーへ送信せず、ブラウザ内で画像生成します。画像生成とあわせて、投稿文コピーや代替テキスト補助も別PRで検討します。

現時点の候補優先度は次のとおりです。

1. `html-to-image`、`modern-screenshot`、`dom-to-image-more` を小規模検証候補にする。
2. `html2canvas` は比較用として確認する。
3. `dom-to-image` はメンテナンス停滞懸念があるため、新規導入候補としては優先度を下げる。
4. サーバー側画像生成は、初期実装では採用しない。

画像化対象は、まず案Bの `.json-preview` + 作成者表記 + 注意書きを別ラッパーにする方向を優先します。

## 今後の実装ステップ案

1. 画像生成ライブラリ候補を比較する。
2. ローカル検証で1候補に絞る。
3. 画像化対象DOMを決める。
4. 生成ボタンを設定パネルの共有セクションに追加する。
5. 生成中・成功・失敗メッセージを追加する。
6. PNGダウンロードを実装する。
7. スマホでの生成可否を確認する。
8. Web Share API連携を別PRで検討する。
9. 投稿文生成・コピーを別PRで追加する。
