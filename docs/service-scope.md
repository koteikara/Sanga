# 年間スケジュールページとニュース連動カレンダーの役割整理

このドキュメントは、京都サンガF.C.の年間スケジュールページと、将来的に検討するニュース連動カレンダーの目的・役割分担・データ連動方針を整理するものです。

両方とも一般サポーター向けのサービスとして設計します。ただし、利用場面と中心機能が異なるため、画面とデータは分けて設計します。

## 1. 年間スケジュールページの目的

年間スケジュールページは、京都サンガF.C.の試合スケジュールをスマートフォンで素早く確認し、閲覧者ごとの観戦予定やメモを保存し、SNSへ共有しやすくするためのページです。

重視することは次のとおりです。

* 一般サポーター向けのスマートフォン最優先ページとして設計する。
* 試合スケジュールを素早く確認できることを重視する。
* 閲覧者ごとに、参戦予定、視聴予定、気になる試合、メモなどを保存できることを重視する。
* LocalStorageを使ったパーソナライズを重視する。
* XなどSNSで参戦予定を共有しやすいことを重視する。
* フルページスクショ用表示は、SNS投稿機能の重要な一部として扱う。
* 試合日程、HOME/AWAY、大会、戦績、チケットURL、放送情報などを見やすく扱う。

年間スケジュールページの中心は、試合そのものです。チケット販売開始日時やイベント申込締切など、公式ニュース記事に紐づく細かな日時情報をすべて同じ画面へ詰め込むのではなく、試合予定の確認、個人ごとの状態管理、SNS共有に集中します。

## 2. ニュース連動カレンダーの目的

ニュース連動カレンダーは、公式ニュース記事に分散している日時情報をカレンダー上で一元確認できるようにする、一般サポーター向けの日時情報一元化サービスです。

重視することは次のとおりです。

* 一般サポーター向けの日時情報一元化サービスとして設計する。
* 公式ニュース記事に分散している日時情報をカレンダー上で確認できるようにする。
* 対象は、チケット販売開始、ファンクラブグレード別販売開始、イベント申込開始、申込終了、イベント開催日時、グッズ販売、キャンペーン締切などとする。
* 利用者は、公開カレンダー上の予定を必要に応じて自分のカレンダーへコピーして使う。
* 自分のカレンダーにコピーすることで、スマートフォンの通知を受け取れる想定とする。
* 年間スケジュールページとは画面・用途を分ける。

ニュース連動カレンダーの中心は、ニュース由来の日時イベントです。試合日程だけでなく、販売開始、申込締切、イベント開催など、見落としやすい日時をカレンダーとして確認できる状態を目指します。

## 3. 2つを分ける理由

年間スケジュールページとニュース連動カレンダーは、どちらもサポーターの行動を助けるものですが、利用目的が異なります。

| サービス | 中心目的 | 主な利用場面 |
| --- | --- | --- |
| 年間スケジュールページ | 試合予定の確認・パーソナライズ・SNS共有 | 年間の試合予定を眺める、参戦予定を整理する、SNSへ共有する |
| ニュース連動カレンダー | ニュース由来の各種日時情報の一元管理 | 販売開始、申込締切、イベント日時を自分のカレンダーへコピーする |

1つの画面に両方を混ぜると、利用目的が曖昧になり、スマートフォンでの見通しと操作性が落ちます。そのため、年間スケジュールページは試合スケジュールと個人状態管理に集中し、ニュース連動カレンダーはニュース由来の日時イベント管理に集中します。

ただし、完全に切り離すのではなく、試合IDなどの共通キーで連動できる余地は残します。たとえば、同じ試合に関連するチケット販売開始イベント、イベント申込締切、関連ニュースURLを将来的に結び付けられるようにします。

## 4. 両サービスで共通化できるデータ

年間スケジュールページの正本は、当面 `public/data/matches.json` とします。ニュース連動カレンダー側は、将来的に `calendar-events.json` のような別データを持つ想定です。

両サービスをまたいで参照できる共通キーとして、`match_id` を使えるようにします。現在のリーグ戦だけを前提に `sec01` 形式へ固定しすぎず、カップ戦やACL戦を見据えて、将来的には `j1-sec01`、`levain-r1`、`acl-md01` のようなID拡張も検討します。

| 項目 | 内部キー候補 | 年間スケジュールページでの扱い | ニュース連動カレンダーでの扱い |
| --- | --- | --- | --- |
| 試合ID | `match_id` / `id` | 試合カードを一意に識別する主キー | 関連する日時イベントを試合へ紐づける外部キー |
| 大会 | `competition` | J1、ルヴァン、ACLなどの内部分類 | イベントが関係する大会の分類 |
| 大会表示 | `competition_label` | 画面表示用の大会名 | カレンダー予定名や説明文の補助 |
| 対戦相手コード | `opponent_code` | CSS、ロゴ、絞り込みなどの補助 | 関連イベントの検索・分類補助 |
| 対戦相手名 | `opponent` | 試合カードの表示名 | カレンダー予定名や説明文の補助 |
| 会場コード | `venue_code` | 将来の会場別表示・絞り込み用 | 現地イベントや会場関連予定の分類補助 |
| 会場名 | `venue` | 試合カードの表示名 | カレンダー予定名や説明文の補助 |
| 関連ニュースURL | `news_url` / `related_news_url` | 試合に関する公式ニュースへの導線 | 日時抽出元や詳細確認先 |
| チケットURL | `ticket_url` | 試合カードからチケット情報へ移動する導線 | 販売開始イベントの詳細URL |
| イベントURL | `event_url` | 試合関連イベントへの導線 | イベント申込開始・締切・開催予定の詳細URL |
| 根拠URL | `source_url` | 試合日程・会場などの根拠 | 日時イベント抽出元の根拠 |
| 確認日 | `source_checked_at` / `checked_at` | 日程データ確認日の記録 | 人間確認済み日時や更新ログの記録 |

## 5. 年間スケジュールページ側の機能優先度

| 優先度 | 機能 | 目的と理由 |
| ---: | --- | --- |
| 1 | データ構造拡張 | 大会、キックオフ時刻、試合状態、チケットURL、放送情報などを扱えるようにし、試合情報の正本としての表現力を上げる。 |
| 2 | パーソナライズ状態の整理 | 参戦予定、視聴予定、気になる試合など、閲覧者ごとの状態を分かりやすく整理する。個人状態はLocalStorage保存を基本にする。 |
| 3 | フィルタリング機能 | HOME/AWAY、大会、状態、参戦予定などで絞り込み、スマートフォンでも必要な試合へ素早くたどり着けるようにする。 |
| 4 | 表示モード切替 | 一覧性重視、詳細確認重視、SNS共有向けなど、利用場面に応じた表示を選べるようにする。 |
| 5 | フルページスクショ用表示 | SNS投稿機能の重要な機能として、テキスト投稿だけでなく、視覚的に分かりやすい参戦予定共有を可能にする。 |
| 6 | SNS投稿補助 | Xなどへ参戦予定を投稿しやすい文言、ハッシュタグ、短縮タイトルを補助し、共有の手間を減らす。 |
| 7 | 追加メモ機能 | 個人メモをLocalStorage中心で保存し、観戦予定や気になる点を端末内で管理できるようにする。 |
| 8 | チケット・観戦準備メモ | 持ち物、移動、チケット購入状況などの観戦準備メモをLocalStorage中心で扱い、個人の準備を支援する。 |
| 9 | カレンダー連携 | 試合日程を個人カレンダーへ追加しやすくする。ただし、ニュース由来の販売開始・締切イベントはニュース連動カレンダー側を基本にする。 |
| 10 | カップ戦・ACL戦追加 | リーグ戦以外も扱えるようにし、`j1-sec01`、`levain-r1`、`acl-md01` などのID拡張を検討する。 |
| 11 | 戦績更新 | 試合後に京都得点、相手得点、結果を反映し、年間の進行状況を確認できるようにする。 |
| 12 | 共有URL機能 | フィルタ条件や表示モードを共有URL化する。個人メモなどLocalStorage内の私的情報は原則含めない。 |
| 13 | 対戦相手別補助情報 | 対戦相手別の補足、会場、関連URLなどを表示し、観戦準備や確認を助ける。 |

パーソナライズ状態は、枠線色だけに依存しない設計へ拡張します。将来的には「参戦予定」「視聴予定」「気になる」などの意味ラベルも検討し、色が識別しにくい利用者にも状態が伝わるようにします。

## 6. ニュース連動カレンダー側の機能優先度

| 優先度 | 機能 | 目的と理由 |
| ---: | --- | --- |
| 1 | 日程イベントデータ設計 | チケット販売開始、申込締切、イベント開催日時などを、試合データとは別の `calendar-events.json` のような構造で管理できるようにする。 |
| 2 | 公式ニュースからの日時候補抽出 | 公式ニュース本文から日時候補を抽出し、登録候補として一覧化する。最初から完全自動登録は目指さない。 |
| 3 | 人間確認フロー | 誤抽出を前提に、抽出結果を人間が確認・修正してから公開カレンダーへ登録する。 |
| 4 | 公開カレンダー登録 | 確認済みの日時イベントを公開カレンダーへ登録し、利用者が必要な予定を自分のカレンダーへコピーできるようにする。 |
| 5 | 更新ログ | いつ、どのニュースから、どの日時イベントを登録・修正したかを追跡できるようにする。 |
| 6 | 年間スケジュールページとの試合ID連動 | `match_id` により、試合に関連するチケット販売やイベント予定を年間スケジュールページ側からも参照できる余地を残す。 |

ニュース連動カレンダーでは、完全自動化よりも正確性と確認可能性を優先します。公式ニュースから候補を抽出し、人間が確認してから登録する設計を基本にします。

## 7. スプレッドシート設計への影響

年間スケジュールページ用の「試合」シートでは、試合カードの表示と試合単位の管理に必要な列を優先します。今後、次の列追加を検討します。

| 追加検討列 | 用途 |
| --- | --- |
| 大会 | J1、ルヴァン、ACLなどの内部分類。 |
| 大会表示 | 画面表示用の大会名。 |
| キックオフ時刻 | 試合開始時刻の表示。 |
| 試合状態 | 未確定、確定、延期、中止、終了などの状態管理。 |
| 京都得点 | 試合後の戦績更新。 |
| 相手得点 | 試合後の戦績更新。 |
| 結果 | 勝敗や引き分けの表示。 |
| チケットURL | 試合単位のチケット情報への導線。 |
| イベントURL | 試合単位で代表的に案内するイベントURL。詳細な日時イベントは別データで管理する。 |
| 放送配信 | DAZN、地上波、配信サービスなどの表示。 |
| 放送配信URL | 放送・配信情報への導線。 |
| 公開メモ | 画面に表示してよい短い補足。 |
| SNS用短縮タイトル | SNS投稿補助やスクショ表示で使う短い試合名。 |
| 表示フラグ | 公開画面に表示するかどうかの制御。 |

一方で、次のようなニュース由来の細かな日時情報は、年間スケジュールページ用の「試合」シートへ入れすぎず、ニュース連動カレンダー側または別の「日程イベント」データとして扱います。

| 別データとして扱う項目 | 理由 |
| --- | --- |
| ファンクラブグレード別販売開始日時 | グレードごとに複数日時が発生し、試合シートが肥大化しやすい。 |
| イベント申込開始日時 | ニュース記事由来の日時イベントとして管理する方が、カレンダー登録と相性がよい。 |
| イベント申込終了日時 | 通知用途が中心であり、ニュース連動カレンダー側で扱う方が分かりやすい。 |
| グッズ販売開始日時 | 試合単位に限らない可能性があり、独立した日時イベントとして扱う。 |
| キャンペーン締切 | 試合予定とは利用目的が異なり、カレンダー通知向けの情報として扱う。 |

スプレッドシート列を完成させる前に、この役割分担に沿って、試合シートへ入れる情報と日程イベントデータへ分ける情報を整理します。
