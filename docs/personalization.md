# パーソナライズ状態と色設定の設計

## この設計の目的

このドキュメントは、年間スケジュールページにおける試合カードのパーソナライズ状態、色枠、利用者ごとのラベル設定、LocalStorage保存方針を整理するものです。

年間スケジュールページでは、試合予定そのものは全閲覧者共通の公開データとして扱います。一方で、どの試合をどのように管理したいかは閲覧者ごとに異なるため、カードの状態、色ごとの意味、将来的な個人メモや観戦準備メモは、公開JSONやスプレッドシートではなく閲覧者の端末内に保存する方針です。

この設計は、今後のフィルタリング、表示モード切替、フルページスクショ用表示、SNS投稿補助、カレンダー連携などの土台として扱います。

## 現在の状態

現行の年間スケジュールページでは、試合カードをタップするとカードの枠線状態が順番に切り替わります。

現在のクリック順は次のとおりです。

```text
未選択 → 赤色枠 → 水色枠 → 未選択
```

現行実装では、試合ごとのカード状態を次のLocalStorageキーに保存しています。

```text
sanga-schedule-button-states-v1
```

保存値は試合IDをキーにしたオブジェクトで、状態は数値として扱います。

```text
0 = 未選択
1 = 赤色枠
2 = 水色枠
```

現在の実装では、`0 / 1 / 2` 以外の値は未選択相当として扱われます。既存利用者の端末に保存済みのLocalStorageデータとの互換性を重視するため、現時点ではこの数値構造を変更しません。

また、現在のJ1リーグ38試合では、試合IDとして `sec01` 〜 `sec38` を使っています。LocalStorageの保存状態はこの試合IDに紐づくため、IDを変更すると閲覧者の保存状態が引き継がれなくなる可能性があります。

## 基本方針

パーソナライズ状態と色設定は、次の方針で扱います。

* ページ側は枠線状態の意味を固定しません。
* 利用者が自分の用途に合わせて、色枠の意味を自由に決められる設計を目指します。
* 現在の赤色枠・水色枠は、初期状態として維持します。
* 既存LocalStorage値 `0 / 1 / 2` との互換性を維持します。
* 将来的には、利用者が色ごとのラベルを設定できるようにします。
* 将来的には、利用者が使う色数や色を選べるようにします。
* スプレッドシートや公開JSONには、閲覧者ごとの状態や意味を保存しません。
* 閲覧者ごとの状態、ラベル、色設定はLocalStorageに保存します。
* 個人メモ、状態ラベル、観戦準備メモ、チケット購入状況、同行者メモなども、全閲覧者共通のスプレッドシートには入れません。

## 状態の意味は利用者が決める

年間スケジュールページでは、色枠そのものに固定の意味を持たせません。

```text
色枠 = 固定された意味ではない
色枠 + 利用者が設定したラベル = パーソナライズ状態
```

たとえば、ある利用者は赤色枠を「参戦予定」として使い、水色枠を「DAZN視聴予定」として使うかもしれません。別の利用者は、赤色枠を「チケット購入済み」、水色枠を「行くか迷っている」として使うかもしれません。

ラベル例は次のとおりです。

| 色 | 利用者によるラベル例 |
| --- | --- |
| 赤 | 参戦予定 |
| 水色 | 気になる |
| 黄 | チケット未購入 |
| 緑 | DAZN視聴予定 |
| 紫 | 遠征検討中 |
| オレンジ | 友人と相談 |
| グレー | 保留 |

これらはあくまで利用者が設定できるラベルの例です。ページ側では、色枠に特定の意味を固定しません。

## 色数と色設定の拡張方針

現時点の状態は次の3状態です。

```text
未選択
赤色枠
水色枠
```

次の拡張候補として、まずはプリセット色を増やす方針を検討します。

```text
未選択
赤
水色
黄
緑
```

将来候補は次のとおりです。

```text
利用者が有効にする色を選べる
利用者が各色にラベルを付けられる
利用者が色の並び順を設定できる
```

いきなり自由なカラーピッカーを導入すると、カード背景とのコントラスト不足、色覚特性による識別しにくさ、スクショ時の見えにくさなどの問題が起きやすくなります。そのため、最初はアクセシビリティを確認したプリセット色から選ぶ方針を推奨します。

プリセット色候補は次のとおりです。

```text
赤
水色
黄
緑
紫
オレンジ
グレー
```

状態数が増えた場合、通常タップだけで状態を順送りすると操作が重くなる可能性があります。

現在のクリック順は次のとおりです。

```text
未選択 → 赤色枠 → 水色枠 → 未選択
```

色数を増やした場合は、たとえば次のようにタップ回数が増えます。

```text
未選択 → 赤 → 水色 → 黄 → 緑 → 未選択
```

そのため、将来的には次のような操作も検討します。

* 通常タップは次の状態へ切替
* 長押しまたは詳細メニューで状態を直接選択
* 状態選択パネルから色を選ぶ
* 設定画面で有効にする色だけを選ぶ

ただし、このドキュメント追加時点では画面実装は行いません。

## LocalStorage保存方針

既存のカード状態保存キーは維持します。

```text
sanga-schedule-button-states-v1
```

将来的な色・ラベル設定用キーとして、次のような別キーを検討します。

```text
sanga-schedule-state-settings-v1
```

保存イメージは次のとおりです。

```json
{
  "states": [
    {
      "value": 1,
      "color": "red",
      "label": "参戦予定",
      "enabled": true
    },
    {
      "value": 2,
      "color": "cyan",
      "label": "気になる",
      "enabled": true
    },
    {
      "value": 3,
      "color": "yellow",
      "label": "チケット未購入",
      "enabled": false
    }
  ]
}
```

保存方針は次のとおりです。

* 試合ごとの状態と、状態設定は別キーで保存します。
* 既存キーは互換性維持のため安易に変更しません。
* 将来キーを変更する場合は、移行処理または事前告知が必要です。
* 試合IDが変わると保存状態が紐づかなくなるため、現行リーグ戦では `sec01` 〜 `sec38` のID維持が重要です。
* 同じ端末でもブラウザが異なる場合は保存状態が引き継がれません。
* ブラウザデータ削除や保存内容削除ボタンで保存状態は消えます。
* 保存内容削除ボタンでは、将来的にパーソナライズ状態と状態設定の両方を削除対象にします。

## アクセシビリティ上の注意

色を利用者が選べるようにする場合でも、色だけに依存しない設計にします。

* 赤色枠・水色枠など、色だけで状態を伝えないようにします。
* 将来的には利用者が設定したラベルをカード内または補助表示として出せるようにします。
* `aria-pressed` だけでなく、状態ラベルを `aria-label` に反映することを検討します。
* スクリーンリーダー利用者にも、現在の状態が分かるようにします。
* フルページスクショ用表示では、色だけでなくテキストやアイコンも併用します。
* プリセット色は、背景やカードデザインとのコントラストを考慮して選びます。
* 色覚特性によって識別しにくい組み合わせを避けます。
* ラベル表示を追加する場合は、スマートフォン画面で試合情報の読みやすさを妨げない位置と大きさにします。

## 今後の機能との関係

| 機能 | パーソナライズ状態との関係 |
| --- | --- |
| フィルタリング | 利用者が設定した状態別に表示できる |
| 表示モード切替 | 状態別のリスト表示や月別表示に使える |
| フルページスクショ用表示 | 利用者が選んだ状態だけを視覚的に共有するために使う |
| SNS投稿補助 | 利用者が選んだ状態の試合一覧を投稿文や画像にする |
| 追加メモ機能 | 状態に加えて1行メモをLocalStorageに保存する |
| チケット・観戦準備メモ | 特定状態の試合に対して購入済み・同行者・交通などを管理する |
| カレンダー連携 | 利用者が選んだ状態の試合だけをカレンダー登録できる |
| 共有URL機能 | 状態そのものは個人保存だが、フィルタ条件の共有に使える可能性がある |

## 画面実装時の注意

このドキュメント追加時点では画面実装は行いません。今後実装する場合は、次の点に注意します。

* 現在のクリック順は当面維持します。
  * 未選択 → 赤色枠 → 水色枠 → 未選択
* 既存LocalStorage値 `0 / 1 / 2` を壊しません。
* 既存の赤色枠・水色枠は当面維持します。
* ラベル表示を追加する場合は、スマートフォン画面で邪魔にならないようにします。
* 保存内容削除ボタンでは、パーソナライズ状態と状態設定の両方を削除対象にします。
* フィルタリング実装時は、非表示になったカードの状態もLocalStorageには残します。
* SNS投稿やスクショ用表示では、利用者が選択した状態に応じて対象試合を抽出できるようにします。
* 色設定を追加する場合は、初期値を用意して設定なしでも既存挙動が維持されるようにします。
* 状態数を増やす場合は、タップ順送りだけでなく直接選択UIも検討します。
* `sec01` 〜 `sec38` など既存IDを変更する場合は、LocalStorage移行の必要性を先に確認します。

## やってはいけないこと

* ページ側で赤色枠・水色枠に特定の意味を固定しないこと。
* 利用者ごとの状態、ラベル、色設定、個人メモ、観戦準備メモをスプレッドシートや公開JSONに保存しないこと。
* 既存LocalStorageキー `sanga-schedule-button-states-v1` を理由なく変更しないこと。
* 既存の `0 / 1 / 2` の保存構造を否定したり、移行なしに破壊したりしないこと。
* 既存の赤色枠・水色枠表示をすぐに廃止する前提で実装しないこと。
* 色だけで状態を伝えるUIにしないこと。
* 状態数を増やすだけで、直接選択UIや設定UIの必要性を無視しないこと。
* 認証情報、サービスアカウントJSON、APIキー、`.env`、Secretsの実値を作成・保存・記載しないこと。
